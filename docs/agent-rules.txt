Products/
  Flora.Social         — композиция соцсети: HTTP-контроллеры, без бизнес-логики
Apps/
  Web                  — Next.js/TS (только *.ts, *.tsx, *.d.ts)

  Desktop, Android, IOS — заготовки
Packages/

Направление зависимостей: **Apps → API → Products → Modules → Infrastructure**.

Запрещено: Modules → Products, Modules → Apps, прямые зависимости между Products, доступ к чужой БД, бизнес-логика в API/Products/Shared/Infrastructure.

## Архитектурные принципы

**Цель:** масштабируемость, чёткие границы модулей, минимальная связность, тестируемость. Новые технологии — только если архитектурно оправданы.

**Структура модуля:** Domain → Application → Infrastructure; Contracts — DTO на границах.

**Коммуникация (порядок предпочтения):**
1. Вызовы внутри модуля
2. Контракты (DTO)
3. gRPC (межмодульно)
4. События (асинхронно)

**Процесс перед кодом:**
1. Какому модулю принадлежит логика?
2. Что нельзя трогать?
3. Что разрешено вызывать?
4. Простейшее решение с сохранением границ
5. Ремонтопригодность важнее краткосрочной скорости

При выборе — **меньше межмодульных зависимостей**, а не меньше кода.

**Антипаттерны:** god services, всё в Shared, дублирование между Products, обход архитектуры «для удобства», quick hacks без объяснения.

**Перед реализацией** — кратко объяснить архитектурное решение. **После** — что сделано, как соблюдены границы, почему безопасно для масштабирования.

**Проверка перед завершением:**
- границы модуля соблюдены?
- зависимости однонаправлены?
- бизнес-логика только в Modules?
- обмен через контракты?
- модуль можно вынести в сервис без большого рефакторинга?

Архитектура: `powershell -ExecutionPolicy Bypass -File .\tools\Validate-Architecture.ps1`

## Git и коммиты

См. `CONTRIBUTING.md`:

- **GPG-подпись** всех коммитов в PR (Verified на GitHub).
- **Conventional Commits:** `тип(область): описание` — `feat`, `fix`, `docs`, `refactor`, `test`, `chore`, `security`, …
- **Без атрибуции Cursor:** не добавлять `Co-authored-by: Cursor`, `Made-with: Cursor`, `--trailer` с упоминанием AI. Автор коммита — только пользователь (`git config user.name` / `user.email`).
- Не коммитить секреты (`.env`, ключи, пароли). Не коммитить без явной просьбы пользователя.
- Не делать force-push в `main` без явного запроса.

## Apps/Web

- Только **TypeScript** (`*.ts`, `*.tsx`, `*.d.ts`); новые `*.js`/`*.jsx` не создавать.
- Общая клиентская криптография FSCP — в `Packages/flora-client-core`; веб дублирует тонкую обёртку в `Apps/Web/lib/fscp/` где нужно.
- **Сетка UI:** координаты `A.B - C.D` — ячейки сетки, не пиксели. Первичная сетка = 15px (`--flora-grid-step`), вторичная = 5px.
  - Верхний край к строке `B`: `top: calc(B * var(--flora-grid-step))`, не `(B-1)*step`.
  - Dashboard: якорь `anchorRow` (часто 9 = низ шапки); линия между `H` и `H+1`: `top: calc((H + 1 - anchorRow) * step)`.
  - Полные правила: `.cursor/rules/apps-web-grid-placement.mdc`, чат: `.cursor/rules/apps-web-messages-chat.mdc`.
- Диагностика «обрезки» под шапкой: сначала z-index/окклюзия, потом overflow; flex-скролл: `flex: 1 1 0%` + `min-height: 0` на пути к scroll-контейнеру.

## Безопасность и приватность

- Уязвимости — **не** в публичные issue; см. `SECURITY.md` (flora.dev.hub@gmail.com).
- FIRA и рекомендации **не** читают plaintext E2E-чатов.
- Известные MVP-ограничения (localStorage ключей на вебе, ratchet, серверная проверка подписи envelope) — в `SECURITY.md`; не маскировать как «готово», если не закрыто.

---

## Дисциплина при неоднозначности

- Выбирать вариант с **большей модульностью**.
- Если не уверен — **спросить**, не гадать.
- Если решение нарушает границы — **отклонить** и предложить альтернативу.
- Минимальный diff: не менять несвязанный код; не over-engineer.
