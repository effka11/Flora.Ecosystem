# FIRA-P — People Recommendations

**Status:** Draft  
**Version:** 0.2  
**Date:** 2026-06-09  
**Depends on:** [`FIRA.md`](./FIRA.md)

---

## Overview

FIRA-P — компонент системы FIRA, отвечающий за рекомендации людей (раздел «Люди»). Его задача — предлагать пользователю потенциально интересных авторов и собеседников, основываясь на социальной близости, пересечении интересов (UIP) и глобальной активности. Компонент строго соблюдает приватность: закрытые аккаунты и заблокированные пользователи всегда исключаются из пула.

---

## Goals & Non-Goals

**Goals:**
- Находить пользователей, близких по интересам и социальному графу.
- Учитывать UIP для рекомендации авторов по темам, а не только по социальным связям.
- Обеспечивать discovery новых/растущих пользователей для здоровья экосистемы.
- Уважать настройки приватности (`UserPrivacySettings`).

**Non-Goals:**
- Рекомендации сообществ (это FIRA-C).
- Поиск пользователей по имени/нику (это отдельный поисковый сервис).
- Предложения в контексте сообщений (это будущий FIRA-DM).

## Architecture Position

**Модуль-владелец:** `Modules/Flora.Users`

```
Flora.Social (HTTP controller)
  └─→ IUserRecommendationService  (Flora.Users.Contracts)
        └─→ UserRecommendationService  (Flora.Users.Application)
              ├─→ IUserRecommendationQueries  (Flora.Users.Infrastructure)
              ├─→ FollowGraphReader  (Flora.Users.Infrastructure)
              └─→ ProfileAccessPolicy  (Flora.Users.Infrastructure)
```

Данные о подписках и граф читаются из `Flora.Users`. Для вычисления `IndividualAffinity` нужен `topicVector` кандидата (темы его публичных постов), который хранится в `Flora.Content`. Этот вектор запрашивается через gRPC-контракт `IContentTopicVectorService` (Flora.Content.Contracts) и передаётся в `FiraContext` как предвычисленная структура — прямого обращения к БД `Flora.Content` из `Flora.Users` нет.

**Механизм (§8.1 FIRA.md):** `Flora.Social` при построении `FiraContext` также вызывает `Flora.Content` для получения `CandidateTopicVectors` — словаря `{userUuid → TopicVector}` для набора кандидатов, и включает его в `FiraRequestMetadata`.

## Algorithm

### Шаг 1 — Генерация кандидатов

| Источник | Начальный вес пула | Описание |
|----------|--------------------|----------|
| Friends-of-friends (2-я степень) | 1.0 | Пользователи, на которых подписаны подписки |
| Члены общих сообществ | 0.7 | Пользователи, состоящие в тех же сообществах |
| Пересечение тем UIP | 0.5 | Пользователи, чей контент близок к UIP (по тегам публикаций) |
| Новые / растущие аккаунты | 0.2 | Аккаунты, зарегистрированные < 30 дней назад с активностью |

Из пула немедленно исключаются (pre-filter, до скоринга):
- Пользователи, на которых уже подписан текущий пользователь.
- Заблокированные пользователи (в обоих направлениях).
- Приватные аккаунты без взаимной подписки (mutual).
- Удалённые и заблокированные платформой аккаунты.

### Шаг 2 — Извлечение признаков

Для каждого кандидата:

- **sharedFollowers** — количество пользователей, на которых подписаны оба (текущий и кандидат).
- **followedByFollowing** — количество подписок текущего пользователя, подписанных на кандидата.
- **topicVector** — вектор тем публичных постов кандидата за последние 90 дней.
- **followerCount** — число подписчиков кандидата.
- **recentPostCount** — число публичных постов за последние 30 дней (индикатор активности).
- **accountAge** — возраст аккаунта в днях.
- **graphDepth** — кратчайшее расстояние в графе подписок. Определение: `depth=1` означает, что хотя бы одна подписка текущего пользователя подписана на кандидата (кандидат — «друг друга»); `depth=2` — общий подписчик есть на расстоянии 2 рёбер. Пользователи, на которых текущий пользователь уже подписан, исключены pre-filter'ом и в граф не попадают.

### Шаг 3 — Скоринг

```
Score(user) = α · IndividualAffinity(user, UIP)
            + β · GlobalRelevance(user)
            + γ · SocialProximity(user, graph)
```

Компонент-специфичные значения весов (Phase 2):

| Параметр | Значение по умолчанию |
|----------|-----------------------|
| `α` (IndividualAffinity) | 0.35 |
| `β` (GlobalRelevance) | 0.25 |
| `γ` (SocialProximity) | 0.40 |
| `DecayLambda` | — (поясн.: FIRA-P не применяет затухание к своим признакам; граф подписок и followerCount не затухают. UIP текущего пользователя затухает отдельно по §2.3 FIRA.md — это происходит независимо от компонента) |

#### IndividualAffinity

```
// Session UIP + Hot Zones (FIRA.md §2.7, §11):
effectiveUip_hotzone[topic] = effectiveUip[topic] × hotZoneFactor(topic, current_user)

IndividualAffinity(candidate) = cosine(topicVector(candidate), effectiveUip_hotzone)   ← ∈ [0, 1]
```

Насколько контент, который публикует кандидат, соответствует интересам текущего пользователя. Оба вектора неотрицательны (`topicVector` — tf-idf/binary, `effectiveUip_hotzone` — нормированный с `clamp to 0` по §2.5 FIRA.md и Hot Zones-усиленный), поэтому косинус гарантированно лежит в `[0, 1]`.

#### GlobalRelevance

```
GlobalRelevance(user) = log10(followerCount + 1) × 2.0
                      + activityBoost(user)
```

```
activityBoost(user) = log10(recentPostCount + 1) × 1.5
```

Глобальная популярность нормируется логарифмом — чтобы крупные аккаунты не доминировали, а активные нишевые авторы имели шанс попасть в рекомендации.

#### SocialProximity

```
SocialProximity(user) = log10(followedByFollowing + 1) × 4.0
                      + log10(sharedFollowers + 1) × 2.0
                      + graphDepthDiscount(graphDepth)
```

```
graphDepthDiscount(depth) = max(0, 1.0 − 0.3 × (depth − 1))
```

Пользователи, на которых подписаны несколько подписок текущего пользователя, получают значительный буст (коэффициент 4.0). Чем дальше кандидат в графе — тем меньше бонус.

### Шаг 4 — Постобработка

- **Diversity:** не более 3 кандидатов из одного источника подряд.
- **New account boost:** аккаунтам младше 30 дней с ≥ 5 публичными постами добавляется аддитивный бонус `newAccountBoost` к итоговому Score для поддержки discovery. Значение `newAccountBoost = config.AdditiveBonusWeight × 0.5` (т.е. пол-масштаба от максимального аддитивного бонуса компонента — не фиксированные 0.5 в единицах Score). Это предотвращает доминирование нового аккаунта над персонализированными кандидатами при любом `AdditiveBonusWeight`.
- **Privacy re-check:** повторная проверка настроек приватности перед финальной выдачей (защита от race conditions при изменении настроек).
- **Dedup:** один и тот же пользователь не появляется в результатах дважды.

### Шаг 5 — Выдача

- Per-user кэш, TTL = 300 с (рекомендации людей менее чувствительны ко времени, чем лента).
- **Триггеры инвалидации** (матрица — [`FIRA.md §13.2`](./FIRA.md)): подписка / отписка пользователя.
- **Метаданные ответа:** `generatedAt` и `expiresAt` (§13.3 FIRA.md).
- Endpoint `has-new` для FIRA-P не реализуется — обновление по TTL (раз в 300 с) достаточно.

## Scoring Formula Summary

```
// --- Session UIP + Hot Zones (FIRA.md §2.7, §11) ---
effectiveUip(u)             = (1 − sessionMomentumWeight) × UIP(u)
                            + sessionMomentumWeight × SessionUip(u)
effectiveUip_hotzone[topic] = effectiveUip[topic] × hotZoneFactor(topic, u)

// --- IndividualAffinity ---
IndividualAffinity(c)       = cosine(topicVector(c), effectiveUip_hotzone)   ← ∈ [0, 1]

// --- GlobalRelevance ---
GlobalRelevance(c)          = log10(followers + 1) × 2.0
                            + log10(recentPosts + 1) × 1.5

// --- SocialProximity ---
SocialProximity(c)          = log10(followedByFollowing + 1) × 4.0
                            + log10(sharedFollowers + 1) × 2.0
                            + graphDepthDiscount(depth)

// --- Итоговый Score ---
Score(c)                    = 0.35 · IndividualAffinity(c)
                            + 0.25 · GlobalRelevance(c)
                            + 0.40 · SocialProximity(c)
```

## Individual vs Global Balance

| Фаза cold start | α | β | γ | Поведение |
|-----------------|---|---|---|-----------|
| Phase 0 (0 сигналов) | 0.0 | 0.45 | 0.55 | Упор на социальный граф и глобальную популярность |
| Phase 1 (1–19) | 0.0 → 0.35 | 0.45 → 0.25 | 0.55 → 0.40 | Линейная интерполяция всех трёх весов; сумма = 1 на каждом шаге |
| Phase 2 (≥ 20) | 0.35 | 0.25 | 0.40 | Полная персонализация |

В Phase 0 социальный граф важнее всего: рекомендовать «людей из окружения» — надёжная стратегия при отсутствии данных о контентных интересах.

## Cold Start Protocol

- **Phase 0:** глобально популярные аккаунты + верифицированные / публичные персоны + наиболее активные новые пользователи платформы.
- **Onboarding acceleration:** если пользователь выбрал темы в Settings → Interests, `IndividualAffinity` рассчитывается немедленно через seeded UIP. Пользователи, чей контент совпадает с выбранными темами, поднимаются даже при `α` в Phase 1.
- **Уточняющий вопрос onboarding:** при первом открытии раздела «Люди» опционально можно показать вопрос «Кого вы ищете?» (технари, художники, игроки...) для быстрого сидирования UIP по типу пользователей.

---

## Feedback Loop

- **Подписка** на рекомендованного пользователя → позитивный сигнал для похожих кандидатов; кандидат исключается из пула.
- **Dismissal** (кнопка «Не интересно» / скрыть) → кандидат не показывается повторно в течение 30 дней; его темы получают лёгкий негативный сигнал в UIP.
- **Блокировка** рекомендованного пользователя → немедленное исключение из всех будущих выдач.

## Privacy Boundaries

Унаследованы из [`FIRA.md §12`](./FIRA.md).

Специфично для FIRA-P:
- Закрытые аккаунты (`isPrivate = true`) исключаются из рекомендаций для всех пользователей, кроме уже взаимных подписок.
- `UserPrivacySettings.DiscoverableByRecommendations` — флаг, позволяющий пользователю полностью отключить появление своего аккаунта в FIRA-P. При `false` аккаунт не попадает в кандидатный пул ни для кого.
- Информация о том, какие подписки текущего пользователя подписаны на кандидата (`followedByFollowing`), используется только для вычисления скора. На клиент передаётся только итоговый список рекомендованных пользователей.

## Open Questions / Future Work

- **Topic vector для пользователей:** ~~архитектурно решено~~ — `IContentTopicVectorService` (Flora.Content.Contracts) через §8.1 FIRA.md. **Pending:** реализация gRPC-контракта и pre-computed materialized view для production-производительности.
- **Mutual-only рекомендации:** стоит ли предлагать закрытые аккаунты как «подайте заявку» vs. вообще не показывать?
- **Long-term dismissal:** 30 дней для dismissal или настраиваемо?
- **Cross-platform users:** если у пользователя несколько устройств — синхронизация dismissed кандидатов.
- **Session UIP в FIRA-P:** насколько полезен Session UIP для рекомендаций людей? Если пользователь только что читал посты о программировании — это повод рекомендовать разработчиков? Возможно, `SessionMomentumWeight` для FIRA-P должен быть ниже, чем для FIRA-F (дефолт: `0.1` вместо `0.2`).
- **Hot Zones применимость:** Hot Zones усиливают тему, но для рекомендаций людей это менее очевидно, чем для ленты. Нужно A/B-тестирование.
