## 2. Криптографическая подпись коммитов — GPG (обязательно)

Все коммиты в Pull Request **должны быть подписаны GPG** (cryptographically signed commits). PR с
неподписанными коммитами **не принимаются**. Это требование обеспечивает подтверждение авторства и
целостность истории.

GitHub должен отображать у ваших коммитов бейдж **«Verified»**.

### Быстрая настройка

```bash
# 1. Сгенерировать ключ (если его нет)
gpg --full-generate-key

# 2. Узнать ID ключа (long form)
gpg --list-secret-keys --keyid-format=long

# 3. Экспортировать публичный ключ и добавить его в GitHub:
#    Settings → SSH and GPG keys → New GPG key
gpg --armor --export <KEY_ID>

# 4. Привязать ключ к git и включить подпись по умолчанию
git config --global user.signingkey <KEY_ID>
git config --global commit.gpgsign true
git config --global tag.gpgsign true
```

Email в ключе GPG **должен совпадать** с email в `git config user.email` и быть подтверждён в вашем
аккаунте GitHub. Отдельный коммит можно подписать вручную флагом `-S`:

```bash
git commit -S -m "feat(auth): добавить проверку TOTP при логине"
```

## 3. Рабочий процесс (Workflow)

1. Сделайте **fork** репозитория и создайте ветку от `main`:
   `git checkout -b feat/<краткое-описание>` или `fix/<краткое-описание>`.
2. Для заметных изменений сначала **откройте Issue** или Discussion, чтобы обсудить подход
   (это экономит время и вам, и нам).
3. Вносите изменения **небольшими, сфокусированными коммитами** с понятными сообщениями.
4. Убедитесь, что проект собирается и проходят проверки (раздел 5).
5. Откройте Pull Request в `main`, заполнив описание: что и зачем меняется.

### Сообщения коммитов

Используйте формат, близкий к Conventional Commits:

```
<тип>(<область>): <краткое описание>

<тело: что и почему, при необходимости>
```

Типы: `feat`, `fix`, `refactor`, `docs`, `test`, `perf`, `chore`, `build`, `security`.

## 4. Архитектурные правила (обязательны к соблюдению)

Flora.Ecosystem имеет строгие границы модулей. Вклад, нарушающий их, будет отклонён.

- Направление зависимостей: **Apps → API → Products → Modules → Infrastructure**.
- **Бизнес-логика — только в `Modules`** (в слоях Domain / Application). `Products` — только
  композиция модулей; `Flora.API` — только маршрутизация, DI, middleware; `Infrastructure` и
  `Flora.Shared` — без бизнес-логики.
- **Запрещено:** прямой импорт внутренних типов одного модуля в другом, доступ к чужой БД,
  межмодульное смешивание логики, прямые зависимости между `Products`.
- Взаимодействие между модулями — только через **контракты (DTO)**, **gRPC** или **события**.
- Для `Apps/Web` используется **только TypeScript** (`*.ts`, `*.tsx`, `*.d.ts`); новые `*.js`/`*.jsx`
  не добавляются.

Подробности — в правилах репозитория (`.cursor/rules/`).

## 5. Проверки перед отправкой PR

Убедитесь, что соответствующая часть проекта собирается и проходит линтинг/типизацию:

- **Backend (.NET):** `dotnet build Flora.Ecosystem.slnx`; `dotnet test tests/Flora.ContractFixtures`.
- **JS (корень):** `npm run ci` (typecheck + lint + test во всех workspaces).
- **Web (`Apps/Web`):** сборка/`lint`/`typecheck` без ошибок; TypeScript-типы строгие.
- **Mobile (`Apps/Mobile`):** `typecheck`/`lint` без ошибок.

Не добавляйте в PR секреты, ключи, `.env`, пароли, IP-адреса серверов и прочие
чувствительные данные. Конфигурация — только через example-шаблоны и переменные окружения.
