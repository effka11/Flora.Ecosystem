# Модель E2E и общей безопасности соцсети FLORA

## Описание

Этот документ описывает целевую production-модель безопасности для сообщений FLORA: настоящее end-to-end encryption, восстановление доступа, recovery-фразу, смену email, заморозку украденного аккаунта, доверенные устройства, API-контракты, rollout и release gates. Защищённый личный чат — протокол **FSCP** (Flora Secure Chat Protocol): нормативная спецификация wire-format, криптографии сообщений и серверной валидации — [FSCP.md](FSCP.md). Этот файл покрывает **платформенный** контур (key epochs, backup/recovery, FSM, trusted devices, freeze).

Главная цель: для пользователя сообщения выглядят как обычный мессенджер, но сервер FLORA не имеет технической возможности расшифровать историю переписки.

Вход осуществляется по email + пароль. На этом фундаменте строится архитектура безопасности, но email и пароль решают разные задачи:

- email подтверждает владение аккаунтным каналом;
- пароль может открыть encrypted key backup;
- recovery-фраза или доверенное устройство могут восстановить E2E material одной или нескольких key epochs;
- сервер не должен иметь fallback-ключа для расшифровки истории.

## Глоссарий

- **E2E root key** — главный пользовательский секрет Messaging **внутри одной key epoch** (у аккаунта может быть несколько root keys в разных epochs). Из него открываются epoch account identity key и conversation/history key wrappers этой же epoch. На сервере не хранится.
- **Password backup** — encrypted key backup, зашифрованный ключом, производным от пароля пользователя. Используется для обычного входа и смены устройства.
- **Recovery backup** — encrypted key backup, зашифрованный ключом, производным от 12 recovery words. Используется только в recovery-flow вместе с email/account verification.
- **Trusted device** — устройство с зарегистрированным device signing key и локальным доступом к E2E material хотя бы одной epoch (в т.ч. к E2E root key активной или восстановленной epoch).
- **E2E locked** — продуктовое состояние «аккаунт доступен, но material нужной epoch на клиенте ещё не восстановлен»; в FSM это обычно сочетание **`locked`** или **`active_new_epoch`** со старой epoch в `KeyEpoch state = locked`. Не смешивать дословно только с узлом `locked`, если аккаунт уже в `active_new_epoch`.
- **Key epoch** — отдельная эпоха E2E-ключей. Если пользователь начинает новые сообщения без восстановления старого root key, создаётся новая epoch, а старая history epoch остаётся locked.
- **Epoch account identity key** — Ed25519 identity key конкретной key epoch. Подписывает device keys этой epoch. В **`unlock-complete`** восстановление/merge доказывается через `epochUnlockSignatures` и массив `epochIdentityPublicKeys` **строго по одному элементу на каждый** `keyEpochId` из `recoveredKeyEpochIds`. В штатном **`PUT .../key-backup`** (состояния **`active`** / **`rotating`**) companion `epochIdentityPublicKeys` покрывает **все** `keyEpochId`, присутствующие в plaintext нового password backup, а не только подмножество «recovered».
- **Freeze** — security-состояние аккаунта при подозрении на угон. Блокирует рискованные Auth и Messaging операции до unfreeze.
- **Key backup revision** — монотонно растущая версия encrypted key backup, защищающая от rollback и lost update.
- **Key-change warning** — предупреждение клиенту, что ключ пользователя или устройства изменился и аутентичность переписки требует внимания.

## Основной принцип

Email восстанавливает доступ к аккаунту, но не расшифровывает сообщения.

Доступ к E2E-истории восстанавливается только через один из пользовательских секретов:

- текущий пароль, если он открывает encrypted key backup нужной epoch;
- recovery-фразу из 12 случайных слов;
- уже доверенное устройство, на котором доступен E2E material нужной epoch.

Если пользователь потерял пароль, recovery-фразу и все доверенные устройства, старая E2E-история должна остаться недоступной. Это не ошибка системы, а обязательное свойство настоящего E2E.

### Абсолютные инварианты восстановления

Эти правила важнее удобства UX и не должны нарушаться в реализации:

- email-подтверждение само по себе не открывает E2E-историю;
- recovery-фраза сама по себе не даёт войти в аккаунт;
- новый пароль после email reset не открывает старый E2E backup, пока E2E material старой epoch не восстановлен через recovery или доверенное устройство;
- после password reset аккаунт входит в состояние **E2E locked**, если новый клиент ещё не получил E2E material старой epoch;
- пока account-level состояние **`locked`** (E2E material старой epoch не восстановлен и пользователь ещё не перешёл в `active_new_epoch` через `POST epochs`), нельзя произвольно перезаписывать key backup, ротировать recovery-фразу, добавлять доверенные устройства **для locked old epoch**, мутировать key material locked epoch и удалять старые encrypted messages; **исключения:** (1) **`POST .../unlock-complete`** в FSM **`recovering`** — единственный путь **восстановить** доступ к старой epoch и заменить password `UserE2EKeyBackup` (новый ciphertext **в теле** `unlock-complete`, отдельный **`PUT .../key-backup` для этой цели не используется**); (2) **`POST .../epochs`** при **`locked`** с полным контрактом Create epoch (новый `UserE2EKeyBackup` **в теле** `POST epochs`, явный `uxConfirmationId` — это **новая** epoch, не восстановление старой истории); прямой **`PUT .../key-backup`** при **`locked`** всегда отклоняется; после перехода **`locked` → `active_new_epoch`** разрешены device pending/approve и сообщения **только для новой** `keyEpochId`, без снятия lock со старых epochs;
- если пользователь без E2E-фактора хочет продолжить переписку, создаётся **новая key epoch** для новых сообщений; старые encrypted messages остаются в старой locked epoch;
- запрещено смешивать сообщения разных key epochs без явного `keyEpochId`;
- если нет старого пароля, recovery-фразы и доверенного устройства, старая история остаётся зашифрованной;
- сервер никогда не должен иметь fallback-ключ для расшифровки истории «по обращению в поддержку» или «по email».

## Что хранит сервер

Сервер может хранить:

- ciphertext сообщений;
- публичные ключи пользователя и устройств;
- зашифрованные key backup и recovery backup;
- индекс **публичных** epoch account identity keys (`KeyEpochPublicIdentity`, см. ниже и `unlock-complete`);
- параметры KDF: salt, version, limits;
- статусы доставки и прочтения;
- список устройств;
- security events;
- состояние заморозки аккаунта;
- pending-заявки на смену email.

Сервер не может хранить:

- plaintext сообщений;
- E2E root key;
- приватные ключи устройств;
- recovery-фразу;
- ключ, которым можно открыть encrypted key backup;
- серверный escrow, позволяющий восстановить историю только по email.

## Модель угроз

### Защищаемся от

- утечки базы данных с сообщениями;
- backend-оператора или взломанного backend, который хочет прочитать переписку;
- восстановления старой истории только через доступ к email;
- использования одной recovery-фразы без доступа к аккаунту;
- перезаписи key backup после угона аккаунта;
- получения новых сообщений отозванным устройством;
- массового offline-чтения истории после компрометации сервера.

### Частично защищаемся от

- подмены публичных ключей сервером или атакующим, контролирующим сервер;
- украденного активного устройства;
- XSS или вредных расширений браузера;
- утечек метаданных: кто кому пишет, когда и как часто.

Эти риски нельзя игнорировать. Для них нужны отдельные механизмы: fingerprint, key transparency, device trust, CSP, минимизация локального хранения ключей, freeze mode и security events.

### Не защищаемся от

- полностью скомпрометированного устройства в момент чтения сообщений;
- пользователя, который сам передал пароль, recovery-фразу и доступ к email;
- вредоносного клиента, которому пользователь доверил свою сессию.

### Compliance, модерация и UGC safety

Инвариант «сервер не расшифровывает историю» **не отменяет** требования продукта к безопасности сообщества и закону. Документ фиксирует **архитектурные вилки**; конкретные юрисдикции и обязательства — только после legal review.

- **Жалоба пользователя.** В тикет уходит: `userUuid`, `conversationUuid` или `postUuid`, `messageUuid` / opaque id контента, категория жалобы, timestamp, **без** plaintext и без recovery/password material. При необходимости расследования модератор работает с **метаданными** и политикой блокировки аккаунта; восстановление содержимого чата **не** выполняется через поддержку без user-held секрета (см. раздел **Support runbook** ниже в этом документе).
- **Незаконный контент (CSAM и аналоги).** Возможные модели (выбрать одну или комбинацию в продукте, зафиксировать в policy): (1) **клиентский** скрининг (hash-matching на устройстве до отправки, блокировка upload) без передачи серверу preimages; (2) **добровольная** передача пользователем отчёта с локально расшифрованным фрагментом в доверенный канал (вне E2E-инварианта «сервер не видит» — явное согласие и отдельный правовой режим); (3) **внешние** сигналы (правоохранение по процедуре) — вне scope этого документа. **Запрещено** декларировать «сервер расшифрует по запросу модерации» как скрытый escrow — это ломает заявленную модель E2E.
- **Спам и злоупотребления без plaintext.** Rate limits, reputation, captcha, ограничение частоты создания объектов, блокировка по **сигналам метаданных** (связка с разделом **Rate limits, cooldown и risk policy**). Серверные классификаторы по **тексту** E2E-сообщений в v1 не применяются; при необходимости ML по контенту — только на клиенте или на явно согласованных не-E2E поверхностях (например публичные посты вне E2E).

## Границы E2E в продукте FLORA

Соцсеть шире, чем 1:1 чат. Ниже — что остаётся **внутри** криптографической гарантии «содержимое сообщений и связанные с ними ключи не доступны серверу в plaintext», а что **обязательно серверное** или **частично приватное**.

### In-scope и out-of-scope для E2E v1

| Область | E2E v1 (содержимое) | Сервер видит (типично) |
| --- | --- | --- |
| Личные сообщения (1:1 / группы по мере готовности) | ciphertext + envelope; plaintext только на клиенте | маршрутизация, размеры, `conversationId`, timestamps, delivery state |
| Лента / посты / комментарии | если заявлено как E2E — тот же класс: **только ciphertext** в хранилище; иначе отдельная политика «публичный пост» (вне E2E) | для E2E-ленты — opaque id, автор uuid, время; **не** plaintext тела |
| Реакции / репосты на E2E-контент | либо ciphertext-обёртка (opaque), либо только счётчики без раскрытия кто именно (политика UX) | агрегаты по политике минимизации |
| Профиль (био, аватар) | по умолчанию **не** E2E — публичный или сервер-controlled контент, если продукт не вводит отдельный «зашифрованный профиль» | явно документировать в продукте |

**Правило:** любая новая соцфича проходит чеклист: «нужен ли серверу plaintext?» Если да — это **не** часть E2E messaging-слоя или требует отдельного opt-in и UX.

### Метаданные соцграфа и лента

- Связи «подписка / друг / участник комьюнити» создают **корреляционный** след (кто с кем контактирует). Политика: минимизировать поля, TTL для исторических связей, опциональные режимы «скрыть список подписок», агрегация активности.
- Согласование с разделом **Observability** (подраздел **Privacy metadata**): лента и уведомления не должны протаскивать **preview** plaintext E2E-сообщений на сервер.

### Поиск и рекомендации без plaintext чатов

- **Server-side search** по содержимому E2E-чатов запрещён (уже зафиксировано). Поиск по **публичным** полям: display name (если не E2E), handles, uuid, хэши тегов, если теги вынесены в индекс без содержимого сообщения.
- **Рекомендации «кого читать»** строятся на графе и engagement-метаданных, не на расшифровке inbox.

### Уведомления и внешние каналы (push, email, SMS)

- **Push (APNs/FCM и т.д.):** только то, что не раскрывает содержимое: «новое сообщение», имя отправителя **по политике** (может быть pseudonym до открытия приложения), `badge`, **без** цитаты текста и без `conversationId` в payload, если это повышает риск корреляции на заблокированном устройстве (настраиваемый уровень).
- **Email / SMS:** одноразовые коды и security-ссылки — как в разделе **Смена email**; не смешивать с телом E2E-сообщения.
- **Корреляция:** провайдер push видит токен и частоту; это **вне** криптографической модели FLORA — пользователю раскрывается в privacy policy.

## Криптографическая модель

### Ключи

Целевая модель разделяет:

- **E2E root key**: главный секрет **конкретной key epoch** для восстановления и синхронизации E2E-состояния этой epoch (не «один root на весь аккаунт навсегда»);
- **epoch account identity key**: долгоживущий ключ личности конкретной E2E epoch;
- **device key**: отдельная пара ключей для каждого браузера или устройства;
- **conversation/session keys**: ключи конкретных диалогов;
- **message keys**: ключи конкретных сообщений или ratchet-шагов.

Сервер хранит только публичные части и encrypted backup.

### Password key backup

При первом включении сообщений клиент генерирует первую key epoch: `keyEpochId`, E2E root key этой epoch и epoch account identity key.

Затем локально:

```text
password + messagingKdfSalt -> passwordWrapKey
passwordWrapKey encrypts epoch-aware E2E material -> UserE2EKeyBackup (ciphertext)
```

`messagingKdfSalt` не является секретом. Он хранится на сервере в Messaging-модуле и нужен только для воспроизводимого KDF.

Нельзя использовать серверный password hash из Auth как E2E-ключ. Auth проверяет пароль, Messaging хранит только отдельный encrypted backup.

### Recovery backup

Recovery-фраза из 12 случайных слов открывает отдельную recovery-копию E2E material для одной или нескольких epochs:

```text
12 recovery words + kdf.saltBase64Url -> recoveryWrapKey
recoveryWrapKey decrypts UserE2ERecoveryBackup (AEAD) -> plaintext keyEpochs[]
```

Recovery-фраза не хранится на сервере.

Сервер хранит запись `UserE2ERecoveryBackup` (см. payload formats), то есть в том числе:

- KDF salt и параметры (`kdf.saltBase64Url`, memory/iterations);
- `ciphertextBase64Url` и AEAD metadata;
- `version`;
- `createdAt` / `updatedAt`;
- `usedAt`.

Сервер не должен хранить `recoveryVerifierHash` или другой проверяемый hash recovery-фразы. Корректность recovery проверяется локальной попыткой расшифровки ciphertext. Это снижает риск offline/online brute force по server-side verifier.

### Recovery-фраза из 12 слов

Требования:

- генерируется только на клиенте через криптографически стойкий random;
- слова выбираются из фиксированного словаря, например 2048 слов;
- 12 слов дают примерно 132 бита энтропии, если слова действительно случайные;
- пользователь не должен придумывать слова сам;
- фраза показывается как аварийный ключ, а не как обычный пароль;
- для применения recovery всегда нужно подтверждение по email;
- сервер не должен иметь endpoint «проверить recovery-фразу»;
- ошибка recovery не должна раскрывать, какое именно слово неверное;
- после нескольких серверных recovery-flow попыток нужен cooldown и security event;
- локальные ошибки ввода слов клиент может ограничивать для UX, но не должен отправлять слова или позиции ошибок на сервер.

### Нормализация recovery-фразы

Чтобы recovery работала одинаково на всех клиентах:

- используется фиксированный wordlist version;
- язык wordlist фиксируется для версии, например `flora-recovery-ru-v1` или `flora-recovery-en-v1`;
- слова нормализуются через Unicode NFC;
- пробелы схлопываются;
- регистр приводится к lower-case, если wordlist case-insensitive;
- порядок слов имеет значение;
- клиент должен уметь подсветить «слово не из словаря», но не должен отправлять позицию неверного слова на сервер;
- checksum допустим только клиентский и не должен превращаться в server-side verifier.

### Почему email + recovery

Recovery-фраза сама по себе не должна логинить в аккаунт.

Email сам по себе не должен расшифровывать историю.

Правильное восстановление:

```text
доступ к аккаунту через email
+
12 recovery words
=
восстановление E2E material доступных key epochs
```

Если злоумышленник знает только recovery-фразу, но не контролирует аккаунт/email, он не должен получить encrypted backup.

Если злоумышленник контролирует email, но не знает recovery-фразу и старый пароль, он не должен открыть старую историю.

## Матрица факторов доступа

В таблице разрешённых операций:

- `Да` — фактор обязателен;
- `Нет` — фактор не используется в этом сценарии;
- `Либо` — достаточно одного из факторов, отмеченных `Либо` в этой строке;
- `Пароль` означает известный текущий пароль до начала flow. Новый пароль после password reset не считается фактором владения, он только становится новым способом открыть future backup после восстановления E2E material нужной epoch.

| Ситуация | Email verify | Пароль | Recovery | Подтверждение доверенным устройством | Результат |
| --- | --- | --- | --- | --- | --- |
| **Обычный вход на уже доверенном устройстве** | Нет | Да | Нет | Нет | Аккаунт и E2E-история доступны |
| **Обычный вход на новом устройстве: стандартный риск** | Да | Да | Нет | Нет | Key backup открывается паролем, устройство становится trusted после email challenge |
| **Обычный вход на новом устройстве: старый email недоступен** | Нет | Да | Нет | Да | Key backup открывается паролем, устройство становится trusted после trusted-device approval |
| **Обычный вход на новом устройстве: высокий риск** | Да | Да | Нет | Да | Требуются пароль, email verify и trusted-device approval |
| **Сброс пароля: единый flow с восстановлением истории** | Да | Нет | Либо | Либо | Аккаунт восстановлен, E2E material старой epoch восстановлен, backup перешифрован под новый пароль |
| **Сброс пароля: единый flow без E2E-фактора** | Да | Нет | Нет | Нет | Аккаунт восстановлен, E2E остаётся locked, старую историю можно восстановить позже |
| **Смена пароля со знанием старого пароля** | Да | Да | Нет | Нет | E2E material доступных epochs открыт старым паролем и перешифрован под новый пароль |
| **Смена email: старый email доступен** | Да | Да | Нет | Нет | Email меняется после cooldown/cancel window |
| **Смена email: старый email недоступен** | Да | Да | Либо | Либо | Новый email подтверждён, email меняется после усиленного cooldown |

Пояснения к таблице:

- для нового устройства стандартный путь — пароль + email verify;
- trusted-device approval может заменить email verify только если старый email недоступен, и только после risk-check;
- при высоком риске trusted-device approval не заменяет email verify, а становится дополнительным фактором;
- для смены email при доступном старом email `Email verify = Да` означает подтверждение старого и нового email;
- для смены email при недоступном старом email `Email verify = Да` означает подтверждение нового email и запуск усиленного cooldown.

Отдельно для инцидентов:

- если `Email доступен = Да` и `Recovery доступна = Да` находятся у атакующего, это критический риск: recovery может открыть историю после account recovery;
- если у атакующего только email, старая E2E-история не открывается;
- если у атакующего только recovery, аккаунт и backup не выдаются;
- если у владельца потеряны пароль, recovery и доверенные устройства, email восстанавливает только аккаунт, но не старую историю.

## Сценарии восстановления

### Обычный вход на новом устройстве

1. Пользователь вводит email/username и пароль.
2. Auth выдаёт limited session до завершения security challenge.
3. Web скачивает encrypted key backup из Messaging.
4. Клиент локально выводит `passwordWrapKey`.
5. Клиент открывает E2E material доступных key epochs.
6. Клиент создаёт device key; устройство становится доверенным только после открытия key backup и свежего email/security challenge.
7. После challenge Auth повышает limited session до full session.
8. История сообщений загружается и расшифровывается локально.

Для пользователя это выглядит как обычный вход в мессенджер.

Limited session до завершения challenge разрешает только минимальные действия:

- получить encrypted password backup после fresh password login;
- создать pending device key;
- завершить email/security challenge;
- отменить вход или включить freeze при подозрении.

Limited session не должна разрешать отправку сообщений, чтение списков диалогов, смену email, ротацию recovery, создание новой epoch, device revoke или любые операции с чужими security settings.

### Смена пароля, когда старый пароль известен

1. Клиент открывает старый encrypted key backup старым паролем.
2. Пользователь подтверждает смену пароля через email по шестизначному коду.
3. Пользователь задаёт новый пароль.
4. Клиент перешифровывает E2E material доступных epochs под новый пароль.
5. Сервер получает новый encrypted key backup.
6. История сохраняется.

### Единый password reset через email

Для пользователя это один мастер восстановления доступа, а не два разных действия «сначала аккаунт, потом чаты». Внутри flow система всё равно разделяет account access и E2E access, но UI ведёт пользователя по одному сценарию.

1. Пользователь выбирает «Забыли пароль?».
2. Пользователь подтверждает владение email.
3. Пользователь задаёт новый пароль.
4. Auth восстанавливает доступ к аккаунту.
5. Messaging временно переводит account-level E2E в **`locked`** (FSM), потому что новый пароль ещё не открывает сохранённый на сервере **ciphertext** старого `UserE2EKeyBackup` и старую key epoch.
6. В том же мастере UI сразу предлагает восстановить историю сообщений одним из способов:
   - ввести recovery-фразу из 12 слов;
   - подтвердить восстановление на уже доверенном устройстве.
7. Если пользователь выбирает recovery:
   - клиент локально выводит `recoveryWrapKey`;
   - клиент локально расшифровывает ciphertext выбранного `UserE2ERecoveryBackup` и получает E2E material доступных epochs;
   - клиент выводит `passwordWrapKey` из нового пароля;
   - клиент перешифровывает E2E material восстановленных epochs в новый `UserE2EKeyBackup` (новый ciphertext под новый пароль).
8. Если пользователь выбирает доверенное устройство:
   - новый клиент создаёт pending-запрос на восстановление E2E-доступа;
   - сервер отправляет уведомление на уже доверенные устройства;
   - старое устройство показывает запрос: новый клиент, примерное место/время, fingerprint нового device key;
   - пользователь подтверждает запрос на старом устройстве;
   - старое устройство передаёт E2E root, conversation material и wrapped epoch account identity material новому устройству через E2E device-to-device envelope, зашифрованный на публичный ключ нового устройства и подписанный ключом доверенного устройства;
   - новый клиент выводит `passwordWrapKey` из нового пароля и перешифровывает E2E material в новый `UserE2EKeyBackup`.
9. Если recovery или trusted-device подтверждение успешны, Messaging снимает **E2E locked**, история снова доступна.
10. Если recovery потеряна и доверенных устройств нет, аккаунт остаётся восстановленным, но старая E2E-история остаётся locked:
    - клиент может предложить пользователю явно начать новую E2E-эпоху для новых сообщений;
    - новая epoch получает отдельный `keyEpochId`, новый E2E root key и отдельный `UserE2EKeyBackup`;
    - UI честно объясняет, что старая история защищена E2E и не может быть восстановлена без recovery-фразы или доверенного устройства;
    - старые encrypted messages не удаляются автоматически: если пользователь позже найдёт recovery-фразу или старое доверенное устройство, историю можно будет восстановить.
11. До успешного восстановления старого E2E-доступа клиент не может ротировать recovery старой epoch, перезаписывать backup старой epoch, добавлять доверенные устройства к старой epoch или удалять старые encrypted messages.

Новая epoch не должна перезаписывать backup старой epoch. Сервер обязан хранить epoch boundaries и запрещать операции, которые меняют key material locked history epoch.

### Recovery-фраза скомпрометирована

1. Пользователь входит в аккаунт на доверенном устройстве.
2. Открывает настройки безопасности.
3. Выбирает «Пересоздать фразу восстановления».
4. Клиент генерирует новые 12 слов.
5. Recovery backup перешифровывается.
6. Старые 12 слов больше не работают.
7. Создаётся security event.

Если есть подозрение, что вместе с recovery-фразой скомпрометирован email или пароль, нужно сначала включить freeze mode, отозвать неизвестные устройства и только затем ротировать recovery.

### Когда пользователь реально теряет старую историю

Потеря старой E2E-истории происходит только если одновременно верны все условия:

- пользователь не знает старый пароль;
- recovery-фраза потеряна;
- нет ни одного доверенного устройства с доступным E2E material нужной epoch;
- encrypted backup на сервере есть, но его нечем открыть.

Если в такой ситуации история всё равно восстанавливается только по email, значит у сервера есть обходной ключ. Это нарушает истинный E2E.

## Доверенные устройства

Вход с нового устройства обязательно сопровождается свежей верификацией по email или другим утверждённым security challenge.

Каждое устройство должно иметь собственный `deviceId` и пару ключей.

Сервер хранит:

- device id;
- публичные ключи устройства;
- дату добавления;
- дату последней активности;
- статус: active, revoked, pending;
- подписи epoch account identity key, если устройство подтверждено.

### Добавление нового устройства

Допустимые способы:

- вход с паролем и открытие encrypted key backup;
- подтверждение со старого доверенного устройства;
- восстановление через email + recovery: после подтверждения аккаунта клиент получает ciphertext recovery backup и локально открывает **только те key epochs, для которых есть соответствующая recovery-запись и верная фраза**.

Новое устройство не должно сразу становиться fully trusted только из-за JWT. Оно получает статус `pending`, пока не выполнит один из допустимых способов подтверждения.

После подтверждения устройство начинает получать новые message envelopes.

### Отзыв устройства

При отзыве:

- устройство перестаёт получать новые сообщения;
- все refresh/session tokens устройства отзываются;
- создаётся security event;
- остальные устройства получают уведомление;
- при высоком риске можно включить freeze mode.

### Подлинность ключей

Шифрование без проверки ключей защищает от чтения базы, но слабее против активного сервера, который может подменить публичный ключ.

Production baseline:

1. **Device key signatures.** Device keys должны быть подписаны epoch account identity key.
2. **Local trust state.** Клиент хранит локально первый увиденный account/device fingerprint для собеседника.
3. **Key-change warnings.** Если у собеседника поменялись ключи, UI показывает предупреждение: «Ключ безопасности изменился».
4. **Safety number / fingerprint.** В диалоге можно сверить короткий fingerprint, QR или код безопасности.
5. **Key transparency — phase 2.** Публичный append-only log ключей нужен позже, чтобы серверу было сложнее незаметно подменять ключи разным пользователям.

Без baseline из пунктов 1-4 E2E остаётся полезным, но не полностью защищает от активной атаки сервера.

## Forward secrecy и ratchet

Минимальный MVP может шифровать сообщения через per-message symmetric key и envelopes для устройств.

Но целевая система должна закладывать:

- `protocolVersion`;
- `senderDeviceId`;
- `conversationId`;
- `keyEpochId`;
- `messageEnvelope`;
- ротацию session keys;
- ratchet-подобный механизм для будущей forward secrecy;
- post-compromise recovery после восстановления контроля над устройством.

Для 1:1 диалогов целевая модель — signal-like протокол: X3DH для установления сессии и Double Ratchet для сообщений.

Для групп в будущем — MLS или sender keys.

Детализация **FSCP v1 / message layer** (состояние 1:1 сессии, RKE, safety number, MVP vs target ratchet, pre-keys): [FSCP.md](FSCP.md).

## Lazy decrypt и загрузка истории

Не нужно запрашивать ключ на каждую порцию истории.

Правильная модель:

1. После входа клиент один раз открывает E2E material (root / conversation state) для всех key epochs, доступных из текущего password backup.
2. Ключи держатся в памяти сессии.
3. Диалоги и сообщения грузятся страницами.
4. Расшифровываются только загруженные страницы.
5. Следующие страницы догружаются и расшифровываются в фоне.

UX:

- список диалогов: первые 10-20 диалогов;
- preview сообщений расшифровываются локально;
- при скролле догружаются следующие диалоги;
- в чате сначала грузятся последние 30-50 сообщений;
- при скролле вверх догружается старая история;
- если расшифровка не успела, UI показывает «Загрузка...», а не криптографический экран.

Так пользователь видит обычный мессенджер, а большие истории не блокируют UI.

## Смена email

Email участвует в восстановлении аккаунта, поэтому смена email — критичная операция.

Смена email не должна автоматически менять E2E-доступ. Даже если email успешно изменён, E2E material остаётся доступен только через пароль, recovery-фразу или доверенное устройство для соответствующей epoch.

### Общие правила email challenge

Любое email-подтверждение в security flow должно быть:

- одноразовым;
- короткоживущим, например 10-15 минут;
- привязанным к конкретному действию: password reset, email change, recovery use, freeze или unfreeze;
- привязанным к аккаунту, request id и risk context;
- недействительным после смены пароля, freeze, завершения flow или превышения попыток;
- не переиспользуемым для другого действия.

Email challenge не должен сам по себе снимать **E2E locked** и не должен разрешать key backup overwrite.

### Смена email, когда старый email доступен

Требовать:

- текущий пароль;
- подтверждение старого email;
- подтверждение нового email;
- свежую авторизацию;
- уведомление на все доверенные устройства;
- security event;
- короткий cooldown для отмены при подозрении;
- freeze/cancel link на старый email, который остаётся действительным до завершения cooldown.

### Смена email, когда старый email недоступен

Требовать усиленный flow:

- текущий пароль;
- recovery-фразу или подтверждение от доверенного устройства;
- подтверждение нового email;
- период ожидания, например 24 часа;
- уведомления на старый email и доверенные устройства;
- возможность быстро включить freeze mode.

Recovery-фраза не должна позволять менять email без пароля или доверенного устройства. Иначе украденная recovery-фраза становится слишком сильным фактором.

Если пользователь не знает пароль, не имеет доступа к старому email и не имеет доверенного устройства, автоматическая смена email запрещена. Такой случай должен уходить в freeze/support recovery с ручной проверкой, без доступа к старой E2E-истории до восстановления E2E material старой epoch.

Если смена email запущена сразу после password reset, аккаунт должен оставаться в усиленном risk state: запрещены key backup overwrite, recovery rotation и добавление устройств до завершения cooldown.

### Ограничения

Во время pending email change нужно блокировать:

- перезапись E2E key backup: **`POST .../unlock-complete`**, **`POST .../unlock-complete/challenge`**, **`POST .../epochs`** с `keyBackup`, штатный **`PUT .../key-backup`**; любые server-side пути, эквивалентные снятию E2E-lock без завершения смены email (см. также таблицу **Email change state** — те же запреты в `pending_*` и `cooldown`);
- ротацию recovery-фразы;
- массовое удаление устройств;
- отключение уведомлений безопасности;
- повторную смену email;
- критичные операции без дополнительного подтверждения.

## Freeze mode

Freeze mode — режим заморозки аккаунта при подозрении на угон.

Freeze должен иметь приоритет над всеми обычными flows. Если аккаунт заморожен, Auth и Messaging обязаны отклонять любые операции, которые могут изменить владение аккаунтом, набор устройств или E2E key material.

### Кто может включить

- пользователь по ссылке «Это были не вы?» из email;
- пользователь с доверенного устройства;
- автоматическая система риска;
- поддержка после ручной проверки.

### Что блокируется

- отправка новых сообщений;
- изменение пароля;
- смена email;
- добавление устройств;
- отзыв устройств, кроме специальных recovery-процедур;
- ротация recovery-фразы;
- перезапись encrypted key backup;
- удаление сообщений и диалогов;
- экспорт данных;
- получение recovery-sensitive backup на новое устройство;
- password reset completion без дополнительной recovery-проверки;
- завершение pending email change.

При freeze password reset может пройти только до proof-of-ownership stage: пользователь может подтвердить email и начать восстановление, но Auth не должен менять пароль, выдавать полноценную новую сессию или снимать E2E locked без unfreeze policy. Завершение password reset в freeze требует дополнительный фактор: recovery-фразу, trusted-device approval или manual review по правилам поддержки.

### Что остаётся доступно

- вход для прохождения восстановления;
- просмотр статуса безопасности;
- чтение уже локально доступной истории на доверенном устройстве, если ключ есть локально;
- связь с поддержкой;
- подтверждение владения аккаунтом.

### Разморозка

Для разморозки нужно сочетание факторов:

- подтверждение email;
- пароль или recovery-фраза;
- доверенное устройство, если доступно;
- cooldown после подозрительных операций;
- отзыв неизвестных сессий и устройств после успешного восстановления.

Если freeze был включён через старый email после спорной смены email, разморозка не должна полагаться только на новый email. Нужен дополнительный фактор: старый пароль, recovery-фраза или доверенное устройство.

## Browser и frontend hardening

E2E в браузере требует строгой защиты клиента.

Обязательные меры:

- строгий Content Security Policy;
- запрет inline scripts;
- Trusted Types для опасных DOM sinks;
- отсутствие plaintext сообщений в logs/analytics;
- запрет third-party scripts на messaging screen;
- dependency pinning и audit для crypto/markdown/html packages;
- не хранить E2E root key в `localStorage`;
- ключи держать в памяти сессии или в IndexedDB только в зашифрованном виде;
- очищать ключи при logout;
- использовать WebCrypto там, где возможно;
- использовать проверенные криптографические библиотеки для X25519, Ed25519, AEAD и KDF;
- не реализовывать криптопримитивы вручную;
- проводить threat review для любых HTML/markdown/rendering features в сообщениях.

## Production crypto specification

Этот раздел фиксирует конкретные технические решения. Реализация не должна выбирать крипто-примитивы «по месту».

### Версия протокола

Все криптографические payload должны иметь версию:

- `e2eProtocolVersion = 1`;
- `backupFormatVersion = 1`;
- `messageEnvelopeVersion = 1`;
- `recoveryFormatVersion = 1`.

Сервер обязан хранить версию рядом с ciphertext и не пытаться интерпретировать plaintext.

Клиент обязан:

- отказываться от неизвестной major-версии;
- поддерживать миграцию minor-версий;
- не смешивать payload разных версий в одном AEAD context.

### Выбранные примитивы

Production stack:

| Назначение | Примитив | Комментарий |
| --- | --- | --- |
| KDF для password/recovery backup | Argon2id | Через проверенную библиотеку, не собственная реализация |
| AEAD для backup и сообщений | XChaCha20-Poly1305 (IETF) | libsodium-совместимая реализация; nonce **192-bit** для XChaCha; для RKE-слоя см. golden [test-vectors/fscp-rke-wrap-key-v1.json](../test-vectors/fscp-rke-wrap-key-v1.json) |
| Device agreement | X25519 | Для device-to-device envelopes |
| Подписи устройств | Ed25519 | Device public keys подписываются **epoch account identity key** соответствующей `keyEpochId` |
| Random | WebCrypto secure random | Вся генерация ключей/nonce/word indexes только через CSPRNG |
| Hash/fingerprint | SHA-256 / BLAKE2b | Только для fingerprint, не для password KDF |

Для протокола v1 fallback на AES-GCM запрещён. Если в будущем нужен AES-GCM режим, он должен быть отдельной версией протокола с отдельными test vectors, nonce rules и migration policy.

### KDF-параметры

Базовые параметры Argon2id:

| Контекст | Memory | Iterations | Parallelism | Salt |
| --- | --- | --- | --- | --- |
| Password backup desktop | 64 MiB | 3 | 1 | 16-32 bytes |
| Password backup low-end/mobile | 32 MiB | 3 | 1 | 16-32 bytes |
| Recovery backup | 64 MiB | 4 | 1 | 16-32 bytes |

Правила:

- параметры хранятся вместе с backup;
- клиент может поднять параметры при ротации backup;
- сервер не должен принимать backup без KDF metadata;
- password KDF и recovery KDF используют разные salt и context string;
- context string включается в KDF/AEAD AAD: `flora.messaging.password-backup.v1` или `flora.messaging.recovery-backup.v1`.

### Иерархия ключей

Минимальная production-иерархия:

```text
E2E root key
  -> opens wrapped epoch account identity private key
  -> conversation history key wrappers
  -> recovery backup encryption material

Device key pair
  -> device identity signing key
  -> device agreement key
  -> local device storage key
```

Правила:

- `E2E root key` не используется напрямую для сообщений;
- сообщения шифруются conversation/session keys;
- root key открывает backup conversation/session keys и epoch account identity material;
- epoch account identity key генерируется как случайная Ed25519 key pair и хранится внутри encrypted backup конкретной `keyEpochId`; он не выводится детерминированно из root key;
- новая key epoch создаёт новый root key и новый epoch account identity key; это должно вызывать key-change warning, потому что старая E2E identity не восстановлена;
- device private keys хранятся локально и не копируются между устройствами;
- при trusted-device approval новое устройство генерирует собственные device keys, а старое устройство передаёт E2E root, conversation material и wrapped epoch account identity material в envelope для нового device public key;
- при отзыве устройства новые envelopes для него не создаются.

## Production payload formats

Все бинарные поля в API передаются как base64url без padding.

### UserE2EKeyBackup

```json
{
  "version": 1,
  "backupRevision": 1,
  "backupKeyId": "uuid",
  "userUuid": "uuid",
  "primaryKeyEpochId": "uuid",
  "epochSetRevision": 1,
  "epochSetHashBase64Url": "...",
  "kdf": {
    "name": "argon2id",
    "memoryKiB": 65536,
    "iterations": 3,
    "parallelism": 1,
    "saltBase64Url": "..."
  },
  "aead": {
    "name": "xchacha20-poly1305",
    "nonceBase64Url": "..."
  },
  "ciphertextBase64Url": "...",
  "createdAt": "utc",
  "updatedAt": "utc"
}
```

AEAD plaintext содержит:

```json
{
  "primaryKeyEpochId": "uuid",
  "keyEpochs": [
    {
      "keyEpochId": "uuid",
      "status": "active|locked|recovered|retired",
      "rootKeyBase64Url": "...",
      "epochAccountIdentityPrivateKeyBase64Url": "...",
      "epochAccountIdentityPublicKeyBase64Url": "...",
      "conversationKeyBackups": [
        {
          "conversationUuid": "uuid",
          "conversationKeyBase64Url": "...",
          "ratchetStateBase64Url": "..."
        }
      ]
    }
  ]
}
```

При записи backup клиент **обязан** выставить `epochAccountIdentityPublicKeyBase64Url` для каждой epoch и проверить, что он соответствует `epochAccountIdentityPrivateKeyBase64Url` (иначе commitment и подписи device keys расходятся с реальностью).

AEAD AAD:

```text
flora.messaging.key-backup.v1 | userUuid | backupRevision | backupKeyId | primaryKeyEpochId | epochSetRevision | epochSetHash | kdfSalt
```

### UserE2ERecoveryBackup

Recovery backup имеет тот же epoch-aware encrypted plaintext, но отдельный KDF salt, отдельный `recoveryRevision`, отдельный `recoveryKeyId`, более строгие KDF-параметры и AAD.

```json
{
  "version": 1,
  "recoveryRevision": 1,
  "recoveryKeyId": "uuid",
  "userUuid": "uuid",
  "primaryKeyEpochId": "uuid",
  "epochSetRevision": 1,
  "epochSetHashBase64Url": "...",
  "wordlist": {
    "id": "flora-recovery-ru-v1",
    "wordsCount": 12
  },
  "kdf": {
    "name": "argon2id",
    "memoryKiB": 65536,
    "iterations": 4,
    "parallelism": 1,
    "saltBase64Url": "..."
  },
  "aead": {
    "name": "xchacha20-poly1305",
    "nonceBase64Url": "..."
  },
  "ciphertextBase64Url": "...",
  "createdAt": "utc",
  "updatedAt": "utc",
  "usedAt": "utc|null"
}
```

AEAD AAD:

```text
flora.messaging.recovery-backup.v1 | userUuid | recoveryRevision | recoveryKeyId | primaryKeyEpochId | epochSetRevision | epochSetHash | recoverySalt
```

Запрещено использовать один ciphertext и один salt одновременно для password backup и recovery backup.

`epochSetHashBase64Url` — **commitment** к полному набору epochs и связанным revision-полям, чтобы нельзя было согласовать «частично правдоподобные» метаданные без связки с identity keys и revision.

Вычисление (клиент после decrypt plaintext, или при шифровании перед `PUT`):

1. Для каждой записи в `keyEpochs[]` взять `keyEpochId`, `status` и `epochAccountIdentityPublicKeyBase64Url` из plaintext (обязательное поле; см. выше — должно соответствовать приватному ключу).
2. Построить canonical JSON (лексикографическая сортировка ключей, UTF-8, правила из раздела Canonical encoding) объекта фиксированной схемы, например:

```json
{
  "schema": "flora.epoch-set-commitment.v1",
  "primaryKeyEpochId": "uuid",
  "keyEpochs": [
    { "keyEpochId": "uuid", "status": "active|locked|recovered|retired", "epochAccountIdentityPublicKeyBase64Url": "..." }
  ]
}
```

Массив `keyEpochs` в commitment **сортируется по `keyEpochId`**.

Для **password backup** в preimage commitment дополнительно включаются поля верхнего уровня backup: `backupRevision`, `backupKeyId`, `epochSetRevision` (числа и uuid — в canonical виде).

Для **recovery backup** в preimage дополнительно включаются: `recoveryRevision`, `recoveryKeyId`, `epochSetRevision`.

Итог: `epochSetHashBase64Url = SHA-256(UTF-8(canonicalJSON(commitmentObject)))` в base64url без padding.

Инварианты:

- после локального decrypt клиент **обязан** пересчитать hash из plaintext + outer revision/id полей и отклонить backup при несовпадении с `epochSetHashBase64Url`;
- hash не содержит root keys и private key material, только публичные identity keys epochs;
- смена только «косметики» без изменения commitment запрещена: `epochSetRevision` не считается валидным, если не изменился commitment (см. `unlock-complete` и server-side checks ниже);
- если это **самый первый** password backup пользователя и на сервере ещё нет предыдущей записи, требование «новый `epochSetHash` отличается от сохранённого» не применяется: достаточно согласованности plaintext, outer metadata и монотонных revision; baseline commitment задаётся этим первым `PUT`.

Правила для нескольких epochs:

- backup может содержать несколько `keyEpochs`, если пользователь создал новую epoch и позже восстановил старую;
- новая epoch никогда не перезаписывает старую locked epoch;
- recovery rotation должна перешифровать все доступные epochs в одном новом recovery backup;
- если старая epoch всё ещё locked и её root key недоступен, recovery rotation новой epoch не должна удалять старый recovery backup до явного отказа пользователя от восстановления старой истории;
- система может хранить несколько recovery backup records: current recovery backup для доступных epochs и legacy recovery backup для locked old epochs;
- legacy recovery backup нельзя перезаписывать или удалять автоматически при rotation новой epoch;
- UI должен объяснять, какие epochs защищены текущей recovery-фразой, а какие остаются восстановимыми только старым recovery/device фактором.

```json
{
  "deviceUuid": "uuid",
  "userUuid": "uuid",
  "keyEpochId": "uuid",
  "displayName": "Chrome on Windows",
  "signingPublicKeyBase64Url": "...",

Подпись `signedByEpochAccountIdentityBase64Url` покрывает:

```

Golden-векторы: [test-vectors/fscp-rke-wrap-key-v1.json](../test-vectors/fscp-rke-wrap-key-v1.json), [test-vectors/fingerprint-v1.json](../test-vectors/fingerprint-v1.json).

### DeviceToDeviceRecoveryEnvelope

Trusted-device recovery использует отдельный envelope, а не message envelope.

```json
{
  "version": 1,
  "recoveryRequestId": "uuid",
  "userUuid": "uuid",
  "sourceDeviceUuid": "uuid",
  "targetDeviceUuid": "uuid",
  "transferredKeyEpochIds": ["uuid"],
  "targetAgreementPublicKeyId": "uuid",
  "ephemeralPublicKeyBase64Url": "...",
  "saltBase64Url": "...",
  "aead": {
    "name": "xchacha20-poly1305",
    "nonceBase64Url": "..."
  },
  "ciphertextBase64Url": "...",
  "sourceDeviceSignatureBase64Url": "..."
}
```

Plaintext содержит только E2E material тех epochs, которые пользователь подтверждает передать:

```json
{
  "keyEpochs": [
    {
      "keyEpochId": "uuid",
      "rootKeyBase64Url": "...",
      "epochAccountIdentityPrivateKeyBase64Url": "...",
      "epochAccountIdentityPublicKeyBase64Url": "...",
      "conversationKeyBackups": []
    }
  ]
}
```

AAD:

```text
flora.messaging.device-to-device-recovery.v1 | recoveryRequestId | userUuid | sourceDeviceUuid | targetDeviceUuid | targetAgreementPublicKeyId | transferredKeyEpochIds
```

`sourceDeviceSignatureBase64Url` подписывает canonical envelope без подписи и привязывает передачу к `recoveryRequestId`, target device fingerprint и списку `keyEpochId`. Сервер не расшифровывает этот payload, но проверяет подпись active source device и challenge binding.

Сервер обязан проверить, что `sourceDeviceUuid` имеет active device binding для каждой epoch из `transferredKeyEpochIds`. Если source device active только в новой epoch, он не может подтверждать передачу старой locked epoch.

### Canonical encoding

Для подписей и AAD нельзя использовать «произвольный JSON.stringify».

Правила:

- canonical JSON с лексикографической сортировкой ключей;
- UTF-8 без BOM;
- base64url без padding;
- даты только ISO-8601 UTC;
- массив `recipients` сортируется по `(userUuid, deviceUuid)`;
- неизвестные поля не включаются в подпись v1 и должны приводить к ошибке в strict validation.

### Nonce и key id rules

Правила обязательны для всех ciphertext:

- каждый ciphertext имеет явный `keyId`, `backupKeyId`, `recoveryKeyId` или `messageKeyId`;
- для `recipientKeyEnvelope` контекст ключа согласования однозначно задаётся AAD строки `flora.messaging.recipient-key-envelope.v1` (см. MessageEnvelope): в практической реализации **уникальность nonce** должна обеспечиваться в паре `(ephemeral agreement key material, nonce)` и во всех полях AAD, включая `recipientAgreementPublicKeyId`, `messageKeyId`, `conversationUuid`, `keyEpochId`, `senderDeviceUuid`, `recipientDeviceUuid`; два envelope с одним и тем же `messageKeyId` и тем же ephemeral для одного получателя запрещены;
- один и тот же nonce не используется дважды с одним и тем же ключом;
- для XChaCha20-Poly1305 nonce генерируется случайно 192-bit через CSPRNG;
- nonce не секретен и хранится рядом с ciphertext;
- AAD всегда включает версию протокола и идентификаторы контекста;
- сервер не должен принимать ciphertext без version, algorithm, nonce и key id;
- клиент не должен decrypt payload, если AAD context не совпал с ожидаемым.

| `keyBackup` | объект `UserE2EKeyBackup` | да |
| `newDeviceSigningPublicKeyBase64Url` | base64url | да |
| `newDeviceAgreementPublicKeyBase64Url` | base64url | да |
| `recoveryUnlockToken` | opaque string | одно из |
| `trustedDeviceApprovalToken` | opaque string | одно из |

Поле «fresh reset session id» в описательных списках выше = **`resetRequestId`** в JSON.

**`POST .../epochs/{keyEpochId}/devices/pending`** (тело)

| Поле | Тип |
| --- | --- |
| `signingPublicKeyBase64Url` | base64url |
| `agreementPublicKeyBase64Url` | base64url |
| `displayName` | string, опционально |

**Несколько recovery backup records.** Сервер хранит одну или несколько строк `UserE2ERecoveryBackup`, различимых по `recoveryKeyId`. `GET recovery-backups` возвращает для каждой записи: `recoveryKeyId`, `recoveryRevision`, `primaryKeyEpochId`, `epochSetRevision`, `epochSetHashBase64Url`, `wordlist.id`, `createdAt`, `updatedAt`, `usedAt`, опционально `coversKeyEpochIds` (если сервер ведёт индекс по последнему успешному decrypt metadata — только uuid epochs, без ключей). Клиент в recovery-flow выбирает запись (обычно последнюю по политике UX или явный выбор пользователя при нескольких фразах), затем запрашивает `GET recovery-backup/{recoveryKeyId}`. `PUT recovery-backup` всегда содержит полный объект; `recoveryKeyId` в теле определяет upsert: существующая запись с тем же id обновляется, новый id создаёт дополнительную запись (например legacy для locked epoch). Сервер не удаляет legacy запись автоматически при `PUT` с новым `recoveryKeyId`.

Запреты:

- `PUT key-backup` запрещён при `freeze = true`;
- `PUT key-backup` при account state **`locked`** **всегда** отклоняется; при **`locked`** новый password `UserE2EKeyBackup` допускается **только** в теле **`POST .../unlock-complete`** (FSM **`recovering`**, восстановление/merge) или в теле **`POST .../epochs`** (новая epoch по контракту Create epoch), но **не** отдельным `PUT`;
- `PUT recovery-backup` запрещён при account state **`locked`**; при **`active_new_epoch`** разрешён `PUT recovery-backup`, если не `freeze` и не `pending_*` email change: только ротация/добавление записей для **текущей** recovery-политики (новая epoch / доступные epochs), без автоматического удаления legacy ciphertext;
- `POST epochs` разрешён при account state **`locked`** (FSM) только после fresh account recovery, явного UX-confirmation и idempotency key; он создаёт новую epoch и не изменяет locked epochs;
- `GET key-backup` требует fresh password login или trusted-device challenge, а не обычный JWT; после password reset сервер может по-прежнему отдавать **тот же** сохранённый ciphertext старого `UserE2EKeyBackup` (KDF под **старый** пароль) — клиент с **новым** паролем локально не откроет его до recovery/`unlock-complete` или до явного **`POST .../epochs`** с новым backup; это ожидаемо, а не «пустой ответ сервера»;
- `GET recovery-backup/{recoveryKeyId}` требует fresh recovery-flow session после email verify и risk-check;
- `GET recovery-backup/*` разрешён только внутри recovery-flow, создаёт security event и не должен кэшироваться клиентом или CDN;
- `GET recovery-backups` (только метаданные, без ciphertext) разрешён при обычной полной сессии для экрана настроек и выбора записи; не подменяет recovery-flow для выдачи ciphertext;
- все `PUT` требуют idempotency key.

### PUT key-backup request body

Тело запроса — JSON с полями:

- `keyBackup`: объект `UserE2EKeyBackup` (как в разделе Production payload formats);
- `epochIdentityPublicKeys` — **обязателен**, если account state `active` или `rotating`: массив `{ "keyEpochId": "uuid", "epochAccountIdentityPublicKeyBase64Url": "..." }`, отсортированный по `keyEpochId`, ровно по одной записи на каждую epoch, присутствующую в plaintext нового backup; клиент обязан скопировать публичные ключи из того же plaintext, что зашифрован в `keyBackup.ciphertextBase64Url` (сервер ciphertext не расшифровывает; несогласованность ловится на следующем `unlock-complete` или клиентской self-check). Сервер атомарно обновляет `KeyEpochPublicIdentity` после успешного commit.

При **`unlock-complete`** те же публичные ключи передаются **внутри** основного контракта `unlock-complete` (поле `epochIdentityPublicKeys` в запросе `unlock-complete`), а не как отдельный `PUT key-backup` до него. Новый password `UserE2EKeyBackup` в **recovery/merge**-flow передаётся **в теле** `unlock-complete`; для этой цели **`PUT .../key-backup` не вызывается**. Штатный **`PUT .../key-backup`** с телом из раздела «PUT key-backup request body» остаётся для ротации в **`active`** / **`rotating`**. Сценарий **новой epoch без восстановления старой** при **`locked`** — отдельный **`POST .../epochs`** с `keyBackup` в теле (см. Create epoch), тоже **не** через `PUT` при **`locked`**.

### Create epoch contract

`POST /api/messaging/e2e/epochs` используется только когда пользователь восстановил аккаунт, но не восстановил старую E2E epoch и явно выбрал продолжить с новой историей.

Запрос обязан содержать поля **`POST /api/messaging/e2e/epochs`** из таблицы **Нормативные поля JSON** выше: `accountRecoverySessionId`, `idempotencyKey`, `uxConfirmationId`, `newKeyEpochId`, `newEpochAccountIdentityPublicKeyBase64Url`, `newDeviceSigningPublicKeyBase64Url`, `newDeviceAgreementPublicKeyBase64Url`, `keyBackup`. Значения `epochSetRevision` и `epochSetHashBase64Url` задаются **внутри** plaintext/commitment объекта `keyBackup`, а не отдельными полями HTTP-тела.

Сервер проверяет:

- аккаунт в account state **`locked`** (FSM);
- новая epoch не совпадает с существующей;
- locked old epochs не удалены из metadata;
- backup revision и epoch set revision монотонно увеличены;
- новый `epochSetHashBase64Url` **отличается** от сохранённого до запроса (в commitment появилась новая epoch), **если** на сервере уже был предыдущий password `UserE2EKeyBackup`; иначе применяется только правило первичного baseline (см. инварианты `epochSetHash`); `epochSetRevision` не увеличивается без изменения commitment (аналогично правилам `unlock-complete`); повтор с тем же idempotency key и идентичным телом разрешён;
- idempotency key не переиспользован с другим payload;
- для `newKeyEpochId` выполняется upsert строки `KeyEpochPublicIdentity` по публичному epoch account identity key из запроса (тот же ключ должен войти в commitment нового backup).

### POST epochs request body (пример)

Тело — JSON с полями верхнего уровня по таблице **Нормативные поля JSON** (ниже — тот же набор, что в скелете API contracts):

```json
{
  "accountRecoverySessionId": "uuid",
  "idempotencyKey": "uuid",
  "uxConfirmationId": "uuid",
  "newKeyEpochId": "uuid",
  "newEpochAccountIdentityPublicKeyBase64Url": "...",
  "newDeviceSigningPublicKeyBase64Url": "...",
  "newDeviceAgreementPublicKeyBase64Url": "...",
  "keyBackup": {}
}
```

Поле `keyBackup` — полный объект **`UserE2EKeyBackup`** в форме из раздела **Production payload formats** (в примере `{}` — заглушка размера).

Публичный ключ новой epoch **должен** совпасть с полем `epochAccountIdentityPublicKeyBase64Url` для `newKeyEpochId` в plaintext `keyBackup` после клиентского decrypt (сервер ciphertext не расшифровывает; несогласованность — ошибка валидации по политике реализации). Отдельного companion-массива `epochIdentityPublicKeys` для `POST epochs` не требуется: индекс обновляется из `newEpochAccountIdentityPublicKeyBase64Url` и из commitment нового backup.

`POST epochs` не считается восстановлением старой истории и не должен снимать locked state со старых epochs.

**Повторный `POST .../epochs` (v1, нормативно).** Успешный create-new-epoch допускается **только** при account state **`locked`** (см. контракт выше). После перехода в **`active_new_epoch`** или **`active`** повторный вызов `POST .../epochs` **запрещён**: сервер **обязан** отклонить запрос с **HTTP 409 Conflict** и стабильным машиночитаемым кодом (рекомендуемое имя: `messaging.e2e.epochs.not_allowed_in_current_account_state`). Тело ошибки — JSON с полями `code`, `message` (человекочитаемое), опционально `accountState`. Расширение политики (вторая «новая» epoch без восстановления старой) допускается **только** в новой версии спецификации (bump `e2eProtocolVersion` / отдельный endpoint), а не вариативно в v1.

### Unlock-complete contract

`unlock-complete` при account state **`recovering`** — единственный разрешённый серверный путь **атомарно заменить** password `UserE2EKeyBackup` и индекс `KeyEpochPublicIdentity` по завершении **восстановления или merge** старой epoch (клиент уже восстановил material локально). Отдельный путь записи нового password backup при **`locked`** без восстановления старой истории — только **`POST .../epochs`** (Create epoch). Для замены через `unlock-complete` клиент передаёт новый `UserE2EKeyBackup` **в теле этого `POST`**, а не через отдельный `PUT /api/messaging/e2e/key-backup`.

**Когда разрешён.** Запрос принимается только при account state **`recovering`** с валидным recovery/merge session. Типичные цепочки: `locked` → `recovering` → `unlock-complete` → `active`; либо **`active_new_epoch` → `recovering`** (пользователь позже нашёл recovery/trusted device для старой epoch) → `unlock-complete` → `active` с merge. Продуктовый термин «E2E locked» в глоссарии означает **недоступность material старой epoch**, а не запрещает вызов из FSM `recovering` после `active_new_epoch`.

Запрос обязан содержать:

- `resetRequestId` (uuid; то же, что «fresh reset session id» в описательном списке);
- idempotency key;
- `challengeId`, выданный через `POST .../unlock-complete/challenge` (или эквивалент) в рамках reset/recovery session;
- `recoveredKeyEpochIds` — см. подраздел **`recoveredKeyEpochIds` (v1, нормативно)** выше; кратко: **≥ 1** uuid для нового запроса, `[]` запрещён (**400**, `messaging.e2e.unlock_complete.recovered_epochs_empty`); idempotent replay — то же непустое тело;
- `epochIdentityPublicKeys` — массив `{ "keyEpochId": "uuid", "epochAccountIdentityPublicKeyBase64Url": "..." }`, отсортированный по `keyEpochId`, ровно по одному элементу на каждый id из `recoveredKeyEpochIds`; значения клиент обязан взять из **того же** локально восстановленного plaintext, что и ключи для подписей;
- `epochUnlockSignatures` — массив объектов `{ "keyEpochId": "uuid", "signatureBase64Url": "..." }`, отсортированный по `keyEpochId`, ровно по одному элементу на каждый id из `recoveredKeyEpochIds`;
- новый `UserE2EKeyBackup` (в JSON поле **`keyBackup`**);
- публичные ключи нового устройства: **`newDeviceSigningPublicKeyBase64Url`**, **`newDeviceAgreementPublicKeyBase64Url`**;
- один из proof tokens:
  - `recoveryUnlockToken`, выданный сервером при старте recovery-flow до локального decrypt и привязанный к reset request;
  - `trustedDeviceApprovalToken`, выданный после подтверждения на active trusted device.

#### `recoveredKeyEpochIds` (v1, нормативно)

- **Новый успешный** `unlock-complete` (не строгий idempotent replay уже принятого тела) **обязан** содержать `recoveredKeyEpochIds` с **≥ 1** uuid.
- **Пустой массив `[]`:** запрещён; сервер отвечает **HTTP 400 Bad Request**, код **`messaging.e2e.unlock_complete.recovered_epochs_empty`**.
- **Idempotent replay:** тот же `idempotencyKey` и **байтово идентичное** JSON-тело (включая тот же непустой `recoveredKeyEpochIds`) — допустимый повтор; семантика не меняется.
- **Согласованность:** множество uuid в `recoveredKeyEpochIds` **должно** совпадать с множеством `keyEpochId` в `epochIdentityPublicKeys` и в `epochUnlockSignatures` (сортировка для canonical подписи — см. строку ниже).
- **Обновление backup без восстановления epoch через `unlock-complete`:** в v1 **не поддерживается**; использовать **`PUT .../key-backup`** в **`active`** / **`rotating`** или иной явно описанный контракт.

Canonical payload для подписи (одинаковый для всех epochs в запросе):

```text
flora.messaging.unlock-complete.v1 | userUuid | resetRequestId | challengeId | backupKeyId | backupRevision | epochSetHashBase64Url | recoveredKeyEpochIds_sorted
```

Для каждого `keyEpochId` из `recoveredKeyEpochIds` клиент подписывает эту строку **epoch account identity private key** соответствующей epoch и помещает подпись в `epochUnlockSignatures`.

Сервер не проверяет recovery-фразу и не знает, успешно ли клиент расшифровал backup, пока клиент не прислал новый backup и proof. Proof model:

- клиент вызывает `POST .../unlock-complete/challenge` (или эквивалент в том же recovery session), получает `challengeId` и привязку к `resetRequestId`;
- при recovery-flow сервер выдаёт короткоживущий `recoveryUnlockToken` после email reset, risk-check и выдачи ciphertext recovery backup;
- клиент локально расшифровывает recovery backup;
- сервер для каждого `keyEpochId` из `recoveredKeyEpochIds` проверяет: (1) Ed25519 `epochUnlockSignatures` относительно **соответствующего** `epochAccountIdentityPublicKeyBase64Url` из `epochIdentityPublicKeys`; (2) если для `(userUuid, keyEpochId)` уже есть строка `KeyEpochPublicIdentity`, переданный публичный ключ **байт-в-байт** совпадает с сохранённым (защита от подмены identity при известном прошлом ключе); если строки ещё нет (например первое восстановление после reset), достаточно (1);
- только если все проверки прошли и множества `keyEpochId` в массивах согласованы, сервер принимает новый encrypted key backup, **атомарно upsert** `KeyEpochPublicIdentity` из `epochIdentityPublicKeys` (и из нового backup для всех epochs, явно присутствующих в новом commitment, по политике реализации) и переводит E2E в `active` (или в состояние с несколькими доступными epochs по политике merge).

Для trusted-device flow:

- новое устройство создаёт approval challenge;
- старое active trusted device подписывает challenge своим device signing key;
- сервер проверяет подпись active trusted device;
- сервер выдаёт короткоживущий `trustedDeviceApprovalToken`, привязанный к `resetRequestId`, `newDeviceUuid`, `approvingDeviceUuid`, `userUuid`;
- старое устройство передаёт новому устройству E2E material, включая wrapped epoch account identity private key, через encrypted device-to-device recovery envelope;
- новый клиент открывает E2E material и формирует `epochUnlockSignatures` так же, как в recovery-flow (те же правила canonical payload и challenge);
- новый клиент отправляет `unlock-complete` с новым encrypted key backup, `epochIdentityPublicKeys`, `trustedDeviceApprovalToken` и `epochUnlockSignatures`.

Сервер проверяет, что:

- `challengeId` выдан текущему flow, не истёк и не был использован для другого успешного `unlock-complete`;
- flow действительно находится в `recovering`;
- proof token связан с текущим user uuid, device uuid, reset request id и истекает быстро;
- `recoveredKeyEpochIds` не пустой; множества `keyEpochId` в `epochIdentityPublicKeys` и в `epochUnlockSignatures` совпадают между собой и с `recoveredKeyEpochIds`;
- **неизменность не восстанавливаемых epochs (клиент + сервер):**
  - клиент после расшифровки старого backup и сборки нового plaintext обязан для каждой epoch, чей `keyEpochId` **не** входит в `recoveredKeyEpochIds`, скопировать объект epoch **byte-for-byte** в том же canonical виде, что и в исходном plaintext (без перестановки полей и без «нормализации», меняющей байты), за исключением запрещённых операций;
  - сервер не расшифровывает backup, но обязан отклонять запрос, если на сервере **уже был** сохранённый password backup и новый outer `epochSetHashBase64Url` **идентичен** предыдущему, а `recoveredKeyEpochIds` не пустой и запрос не является строгим idempotent replay (тот же idempotency key и то же тело): успешное восстановление **должно** менять commitment (статусы/public keys/recovered material для хотя бы одной epoch из списка);
  - сервер обязан отклонять запрос, если **уже был** сохранённый password backup и `epochSetRevision` увеличился относительно него, а `epochSetHashBase64Url` **остался тем же** (запрещённый «пустой» bump revision без изменения набора epochs);
  - повторная отправка с тем же idempotency key и **идентичным** телом (включая hash, revision, ciphertext) разрешена; любое другое сочетание «новый revision / старый hash» без idempotent match — отклоняется;
- trusted-device approval signature валидна, если использовался trusted-device flow;
- key backup revision увеличивается;
- key backup update не происходит повторно с другим payload при том же idempotency key.

### KeyEpochPublicIdentity (серверный индекс)

Таблица `(userUuid, keyEpochId) → epochAccountIdentityPublicKeyBase64Url` (+ опционально `lastBackupRevision`, `updatedAt`). Private keys **не** хранятся.

Индекс обновляется минимум при:

- успешном `POST epochs` — из явного публичного ключа новой epoch в запросе;
- успешном `unlock-complete` — из `epochIdentityPublicKeys` после проверки подписей;
- успешном `PUT key-backup` в **`active`** / **`rotating`** — клиент присылает companion-массив публичных ключей для всех epochs в новом backup; сервер сохраняет снимок как данные от уже аутентифицированного клиента в этих же состояниях (связка с ciphertext без server-side decrypt — ответственность клиента и политики последующих `unlock-complete`).

## State machines

### E2E state

| State | Значение | Разрешено |
| --- | --- | --- |
| `not_initialized` | E2E ещё не включён | init keys |
| `active` | E2E material текущей primary epoch доступен хотя бы одному trusted device, backup валиден | messages, backup rotation, recovery rotation |
| `locked` | аккаунт доступен, но текущий клиент не открыл E2E material старой epoch | recovery/trusted-device unlock или явный переход в новую epoch |
| `active_new_epoch` | новые сообщения работают в новой key epoch, старая history epoch locked | messages только в новой epoch, recovery старой epoch, key-change warning |
| `recovering` | идёт recovery или trusted-device approval | только операции текущего recovery flow: например `GET recovery-backups`, `GET recovery-backup/{id}`, `POST .../unlock-complete/challenge`, **`POST .../unlock-complete`** (с новым `UserE2EKeyBackup` в теле), вспомогательные device/recover-key шаги по политике сессии; **не** штатный `PUT .../key-backup` для смены password backup |
| `rotating` | идёт перешифровка backup/recovery | запрет параллельных rotations |
| `frozen` | аккаунт заморожен | только recovery/unfreeze/status |

Узел **`frozen`** в этой таблице — отображение **account freeze** (Auth / глобальная политика риска) в Messaging: пока freeze активен, mutating Messaging-операции отклоняются по политике freeze mode, а FSM показывает `frozen` (переход `any -> frozen`).

Переходы:

```text
not_initialized -> active
active -> locked       после password reset без E2E material старой epoch
locked -> recovering   recovery words или trusted-device approval
locked -> active_new_epoch  POST epochs после явного UX-confirmation
active_new_epoch -> recovering  пользователь позже нашёл recovery/device старой epoch
recovering -> active   successful rewrap
active -> rotating     key/recovery rotation
rotating -> active     rotation committed
any -> frozen          freeze/risk/support
frozen -> locked       unfreeze Auth без E2E material primary epoch (или без material ни одной epoch по политике)
frozen -> active       unfreeze при наличии E2E material для primary epoch (или по единой зафиксированной политике risk при multi-epoch)
```

При нескольких key epochs продукт **обязан** зафиксировать одно правило: достаточно ли для `frozen -> active` только primary, или любой набор доступных epochs; одно и то же правило применяется в Auth и Messaging, чтобы «нужная epoch» не оставалась неопределённой.

Если старая epoch восстановлена после периода `active_new_epoch`, аккаунт становится `active` с несколькими key epochs. Клиент обязан показывать обе истории как один UX-поток только если каждое сообщение имеет корректный `keyEpochId` и decrypt state для этой epoch.

### KeyEpoch state

Account-level E2E state описывает общий режим аккаунта. Каждая epoch дополнительно имеет собственный state:

| State | Значение | Разрешено |
| --- | --- | --- |
| `active` | root material доступен текущему trusted device | send/read for this epoch, rewrap |
| `locked` | ciphertext и metadata есть, root material недоступен | только recovery/trusted-device unlock |
| `recovered` | epoch была восстановлена после периода locked/new epoch | read, backup merge, optional primary switch |
| `retired` | epoch больше не используется для новых сообщений | read-only, no new sends |

Account `active_new_epoch` означает: новая epoch находится в `active`, одна или несколько старых epochs находятся в `locked`.

При merge нескольких epochs:

- старые и новые сообщения не перешифровываются автоматически;
- новые outgoing messages используют текущую primary epoch;
- восстановленная old epoch остаётся доступной для чтения и может участвовать в rewrap backup;
- device keys получают право на envelopes только для epochs, где device был подтверждён или куда root material был явно передан;
- UI должен маркировать разрыв истории, если между epochs была пауза или смена E2E identity.

### Email change state

| State | Значение |
| --- | --- |
| `none` | нет активной смены |
| `pending_old_email` | ждём подтверждение старого email |
| `pending_new_email` | ждём подтверждение нового email |
| `cooldown` | оба условия выполнены, идёт окно отмены |
| `completed` | email изменён |
| `canceled` | отменено пользователем/freeze/support |
| `blocked` | заблокировано risk/freeze |

Во время `pending_*` и `cooldown` запрещены: любая перезапись password backup — **`POST .../unlock-complete`**, **`POST .../unlock-complete/challenge`**, **`POST .../epochs`** с новым `keyBackup`, штатный **`PUT .../key-backup`**; recovery rotation; массовый revoke устройств; отключение security notifications (нет параллельного «снятия E2E» и смены email в одном окне; согласовано с блоком **Ограничения** при смене email).

## Rate limits, cooldown и risk policy

Production значения по умолчанию:

| Действие | Limit | Cooldown / последствия |
| --- | --- | --- |
| Email code request | 5 в час на аккаунт и IP | далее 1 час cooldown |
| Password reset complete | 3 попытки кода | далее новый code flow |
| Recovery flow start / `GET recovery-backup/{recoveryKeyId}` | 5 в сутки | далее 24 часа cooldown + security event |
| `GET recovery-backups` (только метаданные) | 30 в сутки | отдельный лимит; не выдаёт ciphertext |
| `POST unlock-complete/challenge` | 20 в сутки | привязка к recovery session; не выдаёт ciphertext |
| Trusted-device approval | 10 pending в сутки | далее risk block |
| Key backup overwrite | 5 в сутки | далее manual/risk review |
| Recovery rotation | 2 в сутки | далее 24 часа cooldown |
| Email change | 1 активный flow | cooldown 24 часа перед commit |
| Freeze link | без rate limit для владельца | idempotent, одноразовый token |
| Unfreeze | 3 попытки в сутки | далее manual review |

Строка **Key backup overwrite** — агрегат успешных операций, которые меняют сохранённый password `UserE2EKeyBackup` (штатный **`PUT .../key-backup`**, успешный **`POST .../unlock-complete`**, успешный **`POST .../epochs`** и т.д. по политике продукта). Для алертов и метрик предпочтительно вести **раздельный** учёт путей (см. раздел **Observability без утечки секретов**).

Автоматический freeze рекомендуется при:

- успешном password reset с новой географией + попытка recovery;
- email change сразу после password reset;
- множественные recovery failures;
- множественные отклонённые попытки **`PUT .../key-backup`** при account state **`locked`** (успешный **`unlock-complete`** с новым `UserE2EKeyBackup` в теле не является «обходом через PUT» и в этом триггере не учитывается);
- множественные **отклонённые** или **подозрительно частые** попытки **`POST .../epochs`** при **`locked`** без валидной сессии / `uxConfirmationId` / idempotency (отдельно от PUT-пробинга; успешный легитимный `POST epochs` не должен сам по себе триггерить freeze);
- добавление нескольких устройств за короткий период;
- подтверждение recovery с нового IP/устройства после долгой неактивности.

Сервер не видит каждую локальную ошибку ввода recovery words. Поэтому rate limit применяется к серверным действиям: старт recovery-flow, выдача полного ciphertext recovery backup (`GET recovery-backup/{id}`), неуспешный `unlock-complete`, повторные recovery challenge. Клиент может дополнительно ограничивать локальные попытки ввода, но это UX-защита, а не серверная security boundary.

## Sequence flows

В диаграммах ниже **`Auth -> Messaging: set E2E locked`** означает перевод account-level E2E в FSM **`locked`**; продуктово это же состояние **E2E locked** (см. глоссарий).

### Единый password reset с recovery

```text
User -> Auth: start password reset
Auth -> Email: send one-time code
User -> Auth: code + new password
Auth -> Messaging: set E2E locked
Web -> Messaging: start recovery-flow; optional GET recovery-backups (metadata) to choose recoveryKeyId
Web -> Messaging: GET recovery-backup/{recoveryKeyId} + recoveryUnlockToken
Web -> Messaging: POST unlock-complete/challenge -> challengeId
User -> Web: 12 recovery words
Web: decrypt recovery backup locally
Web: derive wrap key from new password; build new UserE2EKeyBackup with recoveredKeyEpochIds, updated epochSetHashBase64Url and monotonic epochSetRevision
Web: получить challengeId; собрать epochIdentityPublicKeys + epochUnlockSignatures по каждому recovered epoch
Web -> Messaging: unlock-complete with challengeId, recoveredKeyEpochIds, epochIdentityPublicKeys, epochUnlockSignatures, new key backup, recoveryUnlockToken
Messaging: set E2E active
```

### Единый password reset с trusted device

```text
User -> Auth: code + new password
Auth -> Messaging: set E2E locked
NewDevice -> Messaging: create recovery request
Messaging -> OldDevices: notify
OldDevice -> User: show request and fingerprint
User -> OldDevice: approve
OldDevice -> NewDevice: encrypted device-to-device recovery envelope with transferredKeyEpochIds and E2E material for selected epochs
OldDevice -> Messaging: signed approval challenge
Messaging -> NewDevice: trustedDeviceApprovalToken
NewDevice -> Messaging: POST unlock-complete/challenge -> challengeId
NewDevice: build new UserE2EKeyBackup (epochSetHashBase64Url, epochSetRevision, recoveredKeyEpochIds)
NewDevice: epochIdentityPublicKeys + epochUnlockSignatures по каждому recovered epoch (тот же canonical payload)
NewDevice -> Messaging: unlock-complete with challengeId, recoveredKeyEpochIds, epochIdentityPublicKeys, epochUnlockSignatures, new key backup, trustedDeviceApprovalToken
Messaging: set E2E active
```

### Email change with old email

```text
User -> Auth: start email change
Auth -> OldEmail: send confirm/cancel/freeze links
Auth -> NewEmail: send confirm code
User -> Auth: confirm old + new
Auth: enter cooldown
OldEmail/TrustedDevices: can cancel or freeze
Auth: commit after cooldown
```

### Freeze

```text
User/Risk/Support -> Auth: freeze
Auth: revoke risky sessions
Auth -> Messaging: deny key material mutations
Messaging: block sends, backup overwrite, device changes
User -> Auth: unfreeze factors
Auth + Messaging: restore locked or active state
```

## Production acceptance checklist

Перед включением E2E в production должны быть выполнены все пункты:

- plaintext `Content` не записывается для новых сообщений;
- все ciphertext имеют version, nonce, AAD и key id;
- backup и recovery backup используют разные salt и context;
- backup и recovery backup являются epoch-aware и не теряют locked old epochs при создании новой epoch;
- password reset реализован единым мастером, но технически ставит `E2E locked`;
- продуктовое **E2E locked** (недоступность старой epoch) при password reset согласовано с FSM: `locked` / `active_new_epoch`; перезапись password `UserE2EKeyBackup` без штатного `PUT` при **`locked`**: восстановление/merge — только **`POST .../unlock-complete`** (новый backup в теле); явный отказ от восстановления сейчас — **`POST .../epochs`** с новым `keyBackup` в теле; прямой **`PUT .../key-backup`** при **`locked`** отклоняется; recovery rotation и устройства для locked old epoch остаются под запретом до recovery; в **`active_new_epoch`** разрешены device trust и сообщения **только** для новой primary epoch, без снятия lock со старых;
- продолжение переписки без старого E2E-фактора создаёт новую key epoch и не мутирует старую locked epoch;
- создание новой epoch идёт только через отдельный `POST epochs` flow с явным UX-confirmation;
- `unlock-complete` всегда требует валидный `challengeId`, согласованные `epochIdentityPublicKeys` и полный набор `epochUnlockSignatures` по всем `recoveredKeyEpochIds`, даже если root key пришёл через trusted-device flow;
- device-to-device recovery envelope имеет versioned payload, AAD, nonce и challenge binding;
- recipient key envelopes имеют собственный AEAD/AAD и покрыты sender signature;
- message envelope содержит recipient entry для всех active devices участников conversation;
- sender signature выполняется device signing key, а device key подписан epoch account identity;
- recovery attempts имеют rate limit, cooldown и security events;
- email change имеет old/new confirmation, cancel/freeze links и cooldown;
- freeze блокирует Auth и Messaging mutations;
- новое устройство не становится trusted только по JWT;
- операции device trust/approve/revoke/recover-key используют API с явным `keyEpochId` в path (`/epochs/{keyEpochId}/devices/...`);
- key-change warning появляется при изменении device/account keys;
- security events не содержат plaintext, ключи, recovery words или ciphertext bodies;
- события, связанные с E2E epochs, recovery, unlock, device transfer и backup rotation, в метаданных содержат `keyEpochId` или `affectedKeyEpochIds` и при необходимости `epochSetRevision` / `recoveryKeyId` (без секретов);
- frontend не хранит root key в `localStorage`;
- CSP запрещает inline scripts;
- есть test vectors для backup decrypt, recovery decrypt и message decrypt;
- **FSCP v1.0 (формальный релиз спецификации):** соблюдены проверки из [FSCP.md](FSCP.md); в CI или аналоге прогоняются golden-векторы [test-vectors/fscp-rke-wrap-key-v1.json](../test-vectors/fscp-rke-wrap-key-v1.json) и [test-vectors/fingerprint-v1.json](../test-vectors/fingerprint-v1.json);
- реализованы `GET recovery-backups` и `GET recovery-backup/{recoveryKeyId}` с разделением политики (метаданные vs ciphertext);
- реализованы `POST .../unlock-complete/challenge`, поля `epochIdentityPublicKeys` / `epochUnlockSignatures` и индекс `KeyEpochPublicIdentity`;
- есть ручной security review перед релизом;
- **соц-продукт:** зафиксированы границы E2E для ленты/постов/профиля (раздел **Границы E2E в продукте FLORA**), политика жалоб без plaintext, push/email без цитирования E2E-текста, отдельные rate limits или бюджеты на соц-действия при злоупотреблениях;
- **протокол сообщений (FSCP v1):** реализованы требования [FSCP.md](FSCP.md) для v1 (включая handshake и негативный сценарий после revoke device).

## Test vectors и проверяемость

Для production нельзя ограничиваться unit-тестами «шифрует/расшифровывает». Нужны стабильные test vectors, которые будут проходить одинаково в Web, backend validation и будущих клиентах.

Каждый test vector должен фиксировать:

- версию протокола;
- входные ключи и salt в base64url;
- KDF-параметры;
- nonce;
- AAD;
- plaintext;
- ожидаемый ciphertext;
- ожидаемую ошибку для неправильного AAD или неправильного ключа.

Минимальный набор:

| Vector | Что проверяет |
| --- | --- |
| `password_backup_v1_success` | password KDF + decrypt `UserE2EKeyBackup` |
| `epoch_set_commitment_v1` | одинаковый plaintext epochs даёт одинаковый `epochSetHashBase64Url`; изменение `status` или public identity меняет hash; включены `backupRevision`/`backupKeyId` |
| `password_backup_v1_wrong_password` | неправильный пароль не открывает backup |
| `recovery_backup_v1_success` | 12 слов открывают `UserE2ERecoveryBackup` |
| `recovery_backup_v1_wrong_word` | ошибка recovery не раскрывает позицию слова |
| `message_envelope_v1_success` | message key открывает message plaintext |
| `message_envelope_v1_wrong_aad` | изменение `conversationUuid/messageUuid/senderDeviceUuid` ломает decrypt |
| `recipient_key_envelope_v1_success` | recipient envelope открывает message key только нужному device |
| `recipient_key_envelope_v1_wrong_aad` | изменение recipient/user/device/keyEpoch ломает decrypt |
| `fscp_rke_wrap_key_v1_success` | X25519 + HKDF + XChaCha20-Poly1305 IETF → 32-байтовый `messageKey`; данные: [test-vectors/fscp-rke-wrap-key-v1.json](../test-vectors/fscp-rke-wrap-key-v1.json) |
| `fingerprint_v1_success` | safety number 1:1: SHA-256 от UTF-8 preimage; данные: [test-vectors/fingerprint-v1.json](../test-vectors/fingerprint-v1.json) |
| `message_session_revoked_device_v1_failure` | цепочка: handshake/сессия 1:1 → encrypt → decrypt на peer → `POST .../devices/{id}/revoke` отправителя → повторный decrypt или отправка с тем же device id падает по ожидаемой ошибке (см. [FSCP.md](FSCP.md)) |
| `device_to_device_recovery_envelope_v1_success` | trusted device передаёт выбранные epochs новому device |
| `device_to_device_recovery_envelope_v1_wrong_challenge` | изменение recovery request или target device ломает decrypt/signature |
| `device_signature_v1_success` | device key signature валидна |
| `device_signature_v1_tampered_key` | подмена device key обнаруживается |
| `unlock_complete_epoch_unlock_signatures_success` | для каждого `recoveredKeyEpochIds` валидны подпись canonical `flora.messaging.unlock-complete.v1 | ...` и согласованные `epochIdentityPublicKeys` |

Негативные сценарии:

- повтор nonce с тем же key id запрещён validation-тестом;
- backup без version отклоняется;
- backup без KDF metadata отклоняется;
- backup с неверным `epochSetHash` отклоняется после decrypt;
- ciphertext без AAD context отклоняется клиентом;
- прямой `PUT key-backup` при account state **`locked`** (FSM) отклоняется; разрешённая перезапись password `UserE2EKeyBackup` при восстановлении — только через **`POST .../unlock-complete`** из валидного **`recovering`** session (новый backup в теле `unlock-complete`, не отдельный `PUT`); запись нового password backup при **`locked`** на пути «новая epoch без восстановления старой» — только **`POST .../epochs`** с `keyBackup` в теле по контракту Create epoch;
- `PUT recovery-backup` при account state **`locked`** (FSM) отклоняется;
- password reset без recovery/device оставляет историю locked;
- новая переписка после reset без E2E-фактора создаёт новый `keyEpochId`, а не перезаписывает старый backup;
- backup с несколькими epochs восстанавливает каждую доступную epoch независимо;
- recovery rotation не удаляет locked old epoch material без явного отказа пользователя;
- `POST epochs` без `uxConfirmationId` или с удалением locked old epoch metadata отклоняется;
- `POST .../epochs` при account state **не** `locked` (включая `active_new_epoch`) отклоняется с **409** и кодом `messaging.e2e.epochs.not_allowed_in_current_account_state` (v1);
- `unlock-complete` с **`recoveredKeyEpochIds: []`** отклоняется (**400**, `messaging.e2e.unlock_complete.recovered_epochs_empty`);
- `unlock-complete` с изменением неуказанной epoch отклоняется;
- `unlock-complete` с увеличенным `epochSetRevision` при неизменном `epochSetHashBase64Url` (не idempotent replay) отклоняется;
- `unlock-complete` с непустым `recoveredKeyEpochIds` и тем же `epochSetHashBase64Url`, что до запроса, отклоняется (кроме строгого idempotent replay);
- `unlock-complete` без валидного `challengeId`, без полного `epochUnlockSignatures` и без согласованного `epochIdentityPublicKeys` по всем `recoveredKeyEpochIds`, либо с подписью не того canonical payload, отклоняется;
- message envelope без всех active recipient devices отклоняется;
- message envelope со старым `deviceSetRevision` получает conflict и пересобирается клиентом;
- после успешного `POST .../devices/{deviceId}/revoke` для отправителя или получателя клиент и серверная валидация отклоняют дальнейшие операции с отозванным device как с active (соответствует вектору `message_session_revoked_device_v1_failure`);
- freeze блокирует добавление устройств, отправку сообщений и перезапись key backup по политике freeze (глобально), кроме явно разрешённых recovery-процедур;

## Server-side validation contract

Сервер не расшифровывает E2E payload, но обязан валидировать форму и policy.

Сервер обязан проверять:

- `version` присутствует и поддерживается;
- `algorithm` входит в allowlist;
- `nonce` имеет ожидаемую длину;
- `ciphertext` не пустой и не превышает лимиты;
- `keyId/messageKeyId/backupKeyId/recoveryKeyId` присутствует там, где применимо; для `recipientKeyEnvelope` уникальность и контекст проверяются через AAD и связку `messageKeyId` + `recipientAgreementPublicKeyId` + device/uuids;
- `senderDeviceUuid` принадлежит текущему пользователю и active;
- sender device имеет active binding для `keyEpochId`, подписанный epoch account identity key этой epoch;
- recipient device существует и active на момент отправки;
- envelope содержит recipient entry для всех active devices всех участников conversation на момент отправки;
- `keyEpochId` зарегистрирован для пользователя, envelope не смешивает epochs внутри одного ciphertext; отправитель имеет право отправлять в этой беседе с данным `keyEpochId` (участник conversation, не отозван, политика epoch для conversation соблюдена); одна беседа может содержать сообщения с **разными** `keyEpochId` после merge — это норма, а не ошибка;
- `deviceSetRevision` или эквивалентный snapshot участников/устройств совпадает с серверным состоянием на момент отправки;
- `preKeyId = null` для протокола v1, пока полный prekey-протокол не реализован;
- idempotency key присутствует для mutating requests;
- `POST /api/messaging/e2e/epochs` (v1): принимается **только** при account state **`locked`**; иначе — **HTTP 409 Conflict**, код `messaging.e2e.epochs.not_allowed_in_current_account_state` (см. контракт Create epoch);
- `POST /api/messaging/e2e/unlock-complete` (v1): если запрос не является строгим idempotent replay уже принятого тела, `recoveredKeyEpochIds` **обязан** быть непустым; иначе — **HTTP 400 Bad Request**, код `messaging.e2e.unlock_complete.recovered_epochs_empty`;
- политика `freeze` / account-level **`locked`** / смена email (`pending_*`, `cooldown`) **не допускает** операцию, если для данного endpoint и контекста она в документе или product policy запрещена; сервер обязан отклонить запрос с явной ошибкой policy/state (а не «молча» считать состояние нейтральным).

Race condition rule: если состав devices изменился между подготовкой envelope и отправкой, сервер должен вернуть конфликт с актуальным `deviceSetRevision`, а клиент обязан пересобрать recipients list и подпись. Сервер не должен молча принимать envelope без нового active device.

Сервер не должен:

- принимать plaintext fallback для новых клиентов;
- логировать ciphertext bodies целиком;
- логировать recovery attempts с введёнными словами;
- возвращать разные ошибки для «неверная recovery-фраза», «неверный tag», «неверный формат encrypted backup» в пользовательском flow.

### Размерные лимиты

| Объект | Лимит |
| --- | --- |
| Текст сообщения до шифрования | 20 KiB |
| Encrypted message payload | 64 KiB |
| Один recipient envelope | 8 KiB |
| Количество recipient envelopes в 1:1 | до числа active devices двух участников |
| Key backup payload | 2 MiB |
| Recovery backup payload | 2 MiB |
| Device display name | 100 символов |
| Security event metadata | 8 KiB |

Если нужны большие вложения, они не входят в plaintext `body`: файл шифруется отдельным file key, загружается как encrypted object storage payload, а сообщение содержит encrypted attachment descriptor.

## Миграция и rollout

Текущая система уже имеет `UserE2EKey`, `UserMessage`, `EncryptedForReceiver`, `EncryptedForSender` и legacy `Content`. Переход в production E2E должен быть поэтапным.

### Этапы rollout

1. **Read-only compatibility**
   - старые сообщения с `Content` показываются как legacy plaintext;
   - новые клиенты уже умеют читать encrypted envelopes;
   - сервер маркирует legacy messages.

2. **Dual-write запрещён для новых клиентов**
   - новые клиенты отправляют только encrypted payload;
   - `Content` для новых сообщений становится `null`;
   - API отклоняет plaintext для протокола v1.

3. **Device-aware envelopes**
   - вместо одного ключа пользователя используются device keys;
   - сообщения шифруются на active devices участников;
   - revoked devices не получают envelopes.

4. **Recovery и key backup enforcement**
   - пользователь без backup видит мягкий блокирующий экран настройки recovery;
   - после включения recovery появляется production `E2E active`.

5. **Legacy cutoff**
   - отправка plaintext полностью запрещена;
   - чтение legacy plaintext остаётся только для старой истории;
   - миграция старых plaintext сообщений в E2E возможна только клиентом: сервер не должен сам шифровать plaintext в E2E от имени пользователя.

### Правило миграции plaintext истории

Если в базе есть старые plaintext `Content`, сервер уже знает эту историю. Чтобы не создавать ложное обещание, UI должен отличать:

- legacy сообщения, созданные до включения E2E;
- E2E сообщения после включения протокола.

Если пользователь хочет «защитить старую историю», клиент может скачать legacy plaintext, локально зашифровать и отправить encrypted replacement. После этого сервер удаляет plaintext только через отдельный audited migration endpoint. Это не превращает прошлое в настоящее E2E, но уменьшает дальнейший риск хранения plaintext.

### Rollback policy

Rollback не должен требовать расшифровки E2E payload и не должен возвращать plaintext write-path.

Разрешённый rollback:

- временно отключить создание новых E2E conversations;
- оставить чтение уже созданных encrypted messages;
- оставить отправку сообщений только для клиентов, прошедших E2E capability check;
- включить maintenance banner и fallback UX «обновите клиент».

Запрещённый rollback:

- принимать plaintext `Content` от клиентов протокола v1;
- серверно расшифровывать или перешифровывать E2E payload;
- понижать `e2eProtocolVersion` существующей conversation;
- удалять key-change warnings ради совместимости.

## Observability без утечки секретов

Разрешённые метрики:

- количество password reset start/complete;
- количество переходов E2E state: `not_initialized`, `locked`, `active`, `active_new_epoch`, `recovering`, `rotating`, `frozen`;
- количество recovery success/failure;
- количество device add/revoke;
- количество key backup overwrite (желательно раздельно: штатный **`PUT .../key-backup`**, успешный **`unlock-complete`**, успешный **`POST .../epochs`** — для алертов и отличия легитимного трафика от зондирования);
- количество freeze/unfreeze;
- latency decrypt на клиенте без отправки plaintext;
- размер ciphertext payload;
- количество key-change warnings.

Запрещено логировать:

- plaintext сообщений;
- recovery words;
- password/recovery KDF input;
- root key/device private key;
- decrypted backup;
- message plaintext preview;
- полный ciphertext body.

Для debugging допустимы только:

- request id;
- user uuid;
- device uuid;
- protocol version;
- error category;
- ciphertext size;
- короткий hash/fingerprint ciphertext для корреляции без восстановления body;
- security state.

### Privacy metadata

E2E защищает содержимое сообщений, но metadata тоже должна быть минимизирована:

- conversation list возвращает только metadata, необходимую для списка диалогов: participant ids, last message timestamp, unread counters и opaque last-message marker;
- server-side search по plaintext сообщений в E2E v1 запрещён;
- preview последнего сообщения хранится только encrypted и расшифровывается на клиенте;
- attachment file names, mime hints и размеры по возможности шифруются в message plaintext;
- серверу остаются только технические storage-размеры и content-type, нужные для доставки;
- presence/read receipts проектируются отдельно как metadata-sensitive feature и должны иметь отдельные privacy-настройки.

## Release gate

E2E нельзя включать для всех пользователей, пока не выполнены gates:

1. **Cryptography gate**
   - test vectors проходят в Web и backend validation;
   - ручная проверка nonce/key id rules;
   - device-to-device recovery envelope проходит positive/negative vectors;
   - sender device signature проверяется через epoch account identity;
   - dependency audit крипто-библиотек;
   - требования **FSCP v1.0** ([FSCP.md](FSCP.md)) — envelope, RKE unwrap, safety number, негатив после revoke — закрыты тестами, включая golden [test-vectors/fscp-rke-wrap-key-v1.json](../test-vectors/fscp-rke-wrap-key-v1.json) и [test-vectors/fingerprint-v1.json](../test-vectors/fingerprint-v1.json);

2. **Security policy gate**
   - password reset переводит account-level E2E в FSM **`locked`** (продуктово — **E2E locked**, см. глоссарий);
   - в **`locked`** запрещены **произвольные** мутации key material и штатный **`PUT .../key-backup`**; разрешены только контрактные **`POST .../unlock-complete`** (из **`recovering`**, новый `UserE2EKeyBackup` в теле) и **`POST .../epochs`** (новая epoch, `keyBackup` в теле, `uxConfirmationId`);
   - new epoch flow не перезаписывает locked old epoch (metadata и ciphertext старой epoch сохраняются);
   - freeze блокирует Auth и Messaging mutations вне явно разрешённых recovery/unfreeze путей;
   - email change имеет cancel/freeze window и не параллелит `unlock-complete`/перезапись backup (см. Email change state).

3. **UX gate**
   - пользователь понимает recovery-фразу;
   - password reset выглядит как единый flow;
   - потеря истории объясняется честно и без ложных обещаний;
   - key-change warning не выглядит как обычная ошибка;
   - пользователь понимает, какие данные соцсети **не** покрываются E2E (профиль, публичные посты, метаданные графа — по продуктовой политике).

4. **Social / product gate**
   - границы E2E для ленты, постов, реакций и профиля согласованы с командой продукта и privacy policy;
   - модерация и жалобы не требуют server-side plaintext E2E-чатов;
   - уведомления (push и др.) не включают цитаты E2E-сообщений без явного opt-in.

5. **Operational gate**
   - rate limits включены;
   - security events пишутся;
   - алерты на risk events настроены;
   - support runbook готов.

6. **Migration gate**
   - legacy plaintext помечен;
   - новые сообщения не пишут `Content`;
   - backup/recovery storage поддерживает несколько key epochs;
   - rollback не требует расшифровки E2E payload.

## Support runbook

Поддержка не должна иметь инструментов для расшифровки сообщений.

Разрешённые действия поддержки:

- включить freeze после проверки заявки;
- помочь пользователю понять состояние факторов;
- запустить manual/risk review для смены email;
- отозвать явно скомпрометированные sessions/devices по процедуре;
- восстановить контроль над аккаунтом или email только в Auth-boundary;
- объяснить, что без password/recovery/trusted device старая E2E-история недоступна.

Запрещённые действия поддержки:

- запрашивать recovery-фразу целиком;
- принимать recovery-фразу в тикете/чате;
- «восстановить переписку» без пользовательского секрета;
- менять encrypted key backup вручную;
- отключать E2E locked/freeze без второго фактора;
- снимать `E2E locked` или восстанавливать старую key epoch через manual review без user-held E2E factor.

Manual review может восстановить аккаунтный доступ, сменить email по процедуре или снять freeze по Auth-факторам, но не может восстановить E2E-доступ к старой истории. Для старой key epoch всегда нужен password backup, recovery backup или trusted device с E2E material этой epoch.

Отзыв последнего trusted device:

- пользователь может отозвать последнее trusted device только после подтверждения, что password backup и recovery backup существуют и проходят локальную проверку восстановления;
- если recovery backup отсутствует или не проверен, UI должен показать явное предупреждение: отзыв последнего trusted device может сделать старую историю невосстановимой при потере пароля;
- support не должен принудительно отзывать последнее trusted device без freeze/manual review, потому что это может необратимо ухудшить recovery posture пользователя.

Шаблон ответа при потере всех факторов:

```text
Мы можем помочь восстановить доступ к аккаунту, но не можем расшифровать старую историю сообщений.
Сообщения защищены E2E: ключ есть только у вас на устройствах, в пароле или фразе восстановления.
Если вы позже найдёте фразу восстановления или доверенное устройство, история может быть восстановлена.
```

## Рекомендуемый порядок реализации

1. Зафиксировать security policy: email восстанавливает аккаунт, recovery/password/device открывают E2E.

3. Добавить account freeze и security events.
4. Добавить состояние **E2E locked**, **active_new_epoch** и запреты на key material mutations старой epoch.

6. Добавить отдельный create-new-epoch flow после password reset без E2E-фактора.
7. Добавить device keys, epoch account identity и sender/device signature validation.
8. Добавить recovery-фразу из 12 слов и encrypted recovery backup.
9. Добавить device-to-device recovery envelope.
10. Реализовать API устройств в scope `epochs/{keyEpochId}/devices`, `POST .../unlock-complete/challenge`, индекс `KeyEpochPublicIdentity`, server-side validation contract, `deviceSetRevision`, idempotency и правила `epochSetHash` / `unlock-complete` без «пустого» bump `epochSetRevision`.
11. Заменить mock UI сообщений на реальные encrypted conversations.
12. Добавить lazy decrypt и пагинацию.
13. Добавить смену email с old/new confirmation, cooldown и freeze link.
14. Добавить key-change warnings и fingerprints.
15. Подготовить protocol versioning для ratchet.
16. Усилить Web hardening: CSP, безопасное хранение ключей, запрет plaintext logs.
17. Пройти release gates: crypto, security policy, UX, operational, migration.

## Итоговая позиция

Идеальная система Flora Messaging:

- даёт UX обычного мессенджера;
- автоматически открывает историю при входе с паролем;
- позволяет после подтверждения владения аккаунтом (email) восстановить **те key epochs, для которых доступен recovery backup и верные 12 recovery words** (или trusted device с material этих epochs);
- не раскрывает историю при угоне одного email;
- не раскрывает историю при утечке одной recovery-фразы без доступа к аккаунту;
- считает сочетание «email + recovery украдены» критическим инцидентом и снижает риск через cooldown, freeze, уведомления и security events;
- честно теряет старую историю, если пользователь потерял все пользовательские секреты;
- защищает ключи устройств и предупреждает о смене ключей;
- поддерживает freeze mode при угоне;
- не смешивает Auth, Messaging, Products и Web Responsibilities.

Это сохраняет истинный E2E и даёт практическую надёжность без серверного escrow.
